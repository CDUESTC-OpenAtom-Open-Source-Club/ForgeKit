# ForgeKit 需求文档 (Requirements)

> 版本：v0.1 草案 · 状态：初步撰写 · 更新：2026-07

---

## 1. 背景与动机

### 1.1 现状痛点
- **门槛高**：打包发布（deb/rpm/Docker/多架构/上架）要求掌握 Dockerfile、CI YAML、包格式规范、各发行版差异，新手学习曲线陡峭。
- **现有工具不为降门槛优化**：Vector 等工业级平台假设用户已是专家，不会为「让零基础用户上手」做设计。
- **社团普遍卡在最后一公里**：成员「会写代码，不会发布」，项目停在本地，无法形成可分发产物。

### 1.2 时代契机
2026 年 AI Agent 爆发：复杂决策可交给 AI，人只需表达意图。打包发布的知识密集型决策（选格式、配 CI、调依赖、修报错），正是 AI 擅长的领域。

### 1.3 项目定位
**ForgeKit = 已完成的多架构打包底座 + AI Agent 技能层**。
让「会指挥 AI」的人具备完整发布能力。不是再做一个工业平台，是给工业能力套一层 Agent 协同 + 社团低门槛的外壳。

---

## 2. 目标用户

| 用户 | 场景 | 核心诉求 |
|------|------|----------|
| 社团成员（主要） | 把自己的项目打包发布 | 不学打包细节，AI 搞定，边做边会 |
| 独立开发者（次要） | 快速多平台发布 | 降低 CI/打包心智负担 |
| AI Agent（间接用户） | 调用工具完成发布 | 标准化、原子化、可调用的能力 |

---

## 3. 核心问题陈述

> **如何把「多架构打包发布」这件知识密集、易出错的事，降维成「AI 调用几个原子能力」就能完成？**

---

## 4. 功能性需求

### 4.1 底层能力（复用已完成底座，封装暴露）
- **FR-1**：多架构构建（`aarch64` / `x86_64`）
- **FR-2**：多发行版覆盖（EulerOS / CentOS / Ubuntu）
- **FR-3**：多格式打包（deb / rpm / Docker / tar.gz）
- **FR-4**：CI 触发与测试执行

### 4.2 Agent 能力层（本次重点）
- **FR-5**：MCP Server，暴露原子能力供 AI agent 调用
- **FR-6**：能力原子化（每个能力独立、I/O 明确、可被 AI 理解）
- **FR-7**：工程化知识库（打包 know-how、各发行版坑、决策规则），供 AI 选型与排错
- **FR-8**：CI 失败诊断（捕获日志 + 知识库匹配 + 修复建议 / 自动修复）
- **FR-9**：上架发布（推 apt/yum 源、Docker registry、生成安装脚本、Release Note）

### 4.3 协同与教学能力
- **FR-10**：项目模板（Python/Go/TS 打包模板仓库，fork 即用）
- **FR-11**：引导式首次发布（边做边解释，带教学性）

---

## 5. 非功能性需求

- **NFR-1**：能力原子化、可独立调用（AI 友好，非黑盒大平台）
- **NFR-2**：可扩展（新能力可低成本下放）
- **NFR-3**：多架构正确性（aarch64/x86_64 产物可运行）
- **NFR-4**：知识库可维护（社团成员可贡献经验）
- **NFR-5**：私有部署友好（社团私有源 / 私有 registry）

---

## 6. MVP 范围（v0.1）

**只做最小闭环，验证范式**：
- 1 个 MCP Server（TypeScript），暴露 **2 个能力**：`pack_deb` + `build_docker_image`
- 基础知识库（deb 打包要点 + Docker 最佳实践）
- 1 个项目模板（Python）

**验收标准**：在 Claude Code 里说「把这个 Python 项目打成 deb + docker 镜像」，ForgeKit 完成并产出可用产物。

**MVP 明确不做**：rpm、安装脚本、自动上架、CI 自愈、多语言模板、Web UI。

---

## 7. 成功指标

| 指标 | MVP 目标 | 长期目标（12 个月） |
|------|----------|---------------------|
| 原子能力数 | 2 | 12+ |
| Agent 调用成功率 | 60% | 90%+ |
| 社团成员实际使用 | 3 人 | 20+ |
| 交付周期（代码→全平台包） | 小时级 | 分钟级 |
| star | 私有 | 50+ |

---

## 8. 明确不做（边界）

- ❌ Web UI / 可视化平台（MVP 阶段，Agent 接入优先于界面）
- ❌ 支持所有编程语言（聚焦社团在用的 Python/Go/TS）
- ❌ 重做底层引擎（复用已完成底座）
- ❌ 与 Vector 对标工业级覆盖度（不同赛道）

---

## 9. 风险与应对

| 风险 | 应对 |
|------|------|
| AI 调用成功率低 | 能力原子化 + 知识库充分 + 大量测试用例 |
| 社团成员不采用 | 以真实社团项目驱动，边用边迭代 |
| 范围膨胀 | 死守 MVP 边界，按需下放，不提前做 |
| 与工业平台能力重复 | 不重复造，只做「Agent 协同 + 社团场景」的再封装 |
