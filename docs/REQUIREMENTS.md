# ForgeKit Requirements

> 版本：v0.1 规划稿
> 阶段：需求定义与调研规划
> 原则：需求、设计、技术计划通过后，才进入深度开发。

## 1. 项目一句话定义

ForgeKit 是一个面向 AI Agent 的多端打包工具箱，帮助会写代码但不熟悉多端打包的开发者，把项目从源码推进到可直接分发的多端安装包（不上架）。

第一版不做通用 CI/CD 平台，不做企业级发布系统，也不做上架流程工具。第一版只验证一个问题：

> AI Agent 能能不能可靠调用一组本地工具，把一个真实项目打成可直接分发使用的多端安装包（服务器端、移动端、Web→移动端、桌面端）。

**关键区别**：
- ✅ 本地打包，不上架（简化流程）
- ✅ 多端适配（服务器端、移动端、Web→移动端、桌面端）
- ✅ 可直接分发使用（用户拿到安装包即可使用，不需要编译源码）

## 2. 目标用户

| 用户 | 典型场景 | 当前困难 | ForgeKit 价值 |
|------|----------|----------|---------------|
| 学生开发者 | 课程项目、实验项目、社团项目需要多端交付 | 会写代码，但不会写 Dockerfile、Gradle、Xcode、证书签名 | 用 Agent 指令完成多端打包闭环 |
| 社团成员 | 内部工具需要部署到服务器或分享给手机用户 | 项目停在本地运行，缺少多端分发方式 | 形成统一、可教学、可复用的多端打包流程 |
| 独立开发者 | 小工具需要交付 Docker 镜像、deb/rpm、APK、IPA、exe等 | 不想维护复杂多端构建配置和证书管理 | 降低多端打包维护成本 |
| Web开发者 | Web 项目需要打包成移动端可用版本（PWA 或混合应用） | 不知道如何把 Web 项目适配到移动端 | Web→移动端一键打包能力 |
| AI Agent 用户 | 已经用 Claude Code/Cline/Cursor 写代码 | Agent 会改代码，但缺少可靠多端打包工具 | 给 Agent 标准化多端工具接口和决策依据 |

## 3. Agent 接入与使用方式

ForgeKit 不是只解决“所有平台打包”的问题，更重要的是解决“Agent 如何可靠使用打包能力”的问题。

v0.1 的接入策略是：

| 接入方式 | 定位 | v0.1 决策 |
|----------|------|-----------|
| MCP Server | 主协议，给 Codex、Claude Code、Cline、Cursor 等 Agent 调用工具 | 必做 |
| Agent Skill / Markdown Guide | 辅助说明，告诉 Agent 如何规划、确认、调用和总结 | 必做文档，不先做运行时 |
| `Forge.md` | 项目级打包计划文件，记录目标、策略、风险和结果 | 必做 |
| 自研协议 | 未来可能存在的高级打包语义协议 | v1.0 前不做 |

使用流程：

1. Agent 读取项目。
2. Agent 生成或更新 `Forge.md`。
3. 用户确认打包目标。
4. Agent 调用 ForgeKit MCP 工具执行构建。
5. ForgeKit 将结果和风险写回打包文档。

详细协议规划见 [AGENT_INTEGRATION.md](./AGENT_INTEGRATION.md)，打包文档规范见 [PACKAGING_DOCUMENT.md](./specs/PACKAGING_DOCUMENT.md)；项目框架规范与适配见 [PROJECT_FRAMEWORK.md](./specs/PROJECT_FRAMEWORK.md)。

## 4. 核心问题

很多初中级开发者的项目失败点不在代码本身，而在“最后一公里”：

- 不知道项目应该交付 Docker 镜像、系统包、压缩包还是安装脚本。
- 不知道目标系统、架构、glibc、Python/Node/Go 版本会影响产物。
- 不知道构建失败后应该改代码、改 Dockerfile、改依赖还是改发布目标。
- AI Agent 能写代码，但如果没有明确工具接口，发布步骤仍然不可控。

ForgeKit 的核心问题陈述是：

> 如何把发布工程中的重复知识、系统差异和构建步骤，整理成 Agent 可调用、用户可理解、结果可验证的本地工具链？

## 5. 替代方案与缺口

| 方案 | 能解决什么 | 缺口 |
|------|------------|------|
| GitHub Actions / GitLab CI | 自动化流水线 | 新手仍要写 YAML、Secrets、构建矩阵和产物上传逻辑 |
| Dockerfile | 容器化交付 | 不解决 deb/rpm、安装包、系统服务、发布解释 |
| Buildpacks / Nixpacks | 自动生成镜像 | 偏 Docker-first，Agent 可解释性和系统包能力不足 |
| fpm / nfpm | 快速生成系统包 | 偏 CLI 专家工具，需要用户理解包格式和依赖 |
| GoReleaser | Go 项目发布强 | 语言场景较窄，不覆盖 Agent 编排和教学路径 |
| Claude Code / Cline / Cursor | 代码生成和修改 | 缺少稳定、可验证、面向发布工程的工具契约 |

ForgeKit 的差异点不在于“重新发明打包工具”，而在于：

1. 面向 AI Agent 暴露清晰工具契约。
2. 将打包知识、系统适配和错误诊断沉淀成可检索知识。
3. 对新手友好，输出决策依据，而不是只输出命令。
4. 本地优先，不要求一开始搭建云平台或企业 CI。

## 6. 商业价值判断

ForgeKit 的潜在商业价值来自三个方向：

| 方向 | 价值 | 成立条件 |
|------|------|----------|
| 教学与社团 | 降低工程化教学门槛，形成课程/社团基础设施 | 文档和模板必须足够清楚，能服务真实项目 |
| 独立开发者工具 | 减少小项目发布维护成本 | v0.2 后要支持真实项目失败诊断和产物归档 |
| Agent 工具生态 | 成为 AI Agent 的发布能力插件 | MCP 工具必须稳定、可解释、可组合 |
| 项目级打包文档 | 形成可复用的交付规范 | `Forge.md` 必须能被人和 Agent 同时理解 |

短期商业目标不是收费，而是验证使用需求：

- 是否有人愿意把真实项目交给 ForgeKit 打包。
- 是否能明显减少从“代码可运行”到“产物可交付”的时间。
- 是否能提升 Agent 完成发布任务的一次成功率。

## 7. 需求调研计划

进入深度开发前，需要完成最低限度需求验证。

### 6.1 访谈对象

| 类型 | 数量 | 要求 |
|------|------|------|
| 社团/学生开发者 | 5 人 | 至少有一个需要部署或分享的项目 |
| 独立开发者 | 3 人 | 有 Docker、Release 或安装包经验 |
| AI Agent 用户 | 2 人 | 使用过 Claude Code/Cline/Cursor 做项目 |

### 6.2 访谈问题

1. 你最近一次需要交付项目是什么时候？最后交付了什么？
2. 你现在如何构建 Docker 镜像、安装包或 Release？
3. 哪一步最容易失败或最耗时间？
4. 你是否需要 deb/rpm，还是 Docker 已经足够？
5. 如果 AI Agent 帮你发布项目，你最担心什么？
6. 你更希望用 MCP 工具、CLI 命令，还是 Markdown/Skill 文档引导？
7. 你能接受项目里新增 `Forge.md` 作为打包计划吗？
8. 你愿意提供一个真实项目做 v0.1 验证吗？

### 6.3 通过标准

| 验证项 | 最低标准 |
|--------|----------|
| 真实痛点 | 10 个访谈对象中至少 5 个最近一年遇到发布/打包问题 |
| 试点项目 | 至少 3 个真实项目愿意参与 v0.1 验证 |
| MVP 接受度 | 至少 3 个用户接受 v0.1 只做 Python + Docker/deb 的窄范围 |
| 替代方案缺口 | 能明确说出现有工具为什么对目标用户不够友好 |

**已拍板（v0.0 退出）**：v0.1 采用「Docker 硬闭环 + deb 可选」。若后续试点证明用户只需 Docker，deb 直接后置到 v0.2，不做硬验收。

## 8. v0.1 MVP 定义

### 8.1 MVP 目标

验证服务器端最小闭环（多端路线的第一阶段）：

> 用户通过 AI Agent 说明打包目标，Agent 调用 ForgeKit，本地生成可验证的服务器端产物（Docker 镜像；deb 按需可选），并解释为什么这么构建。

**多端路线总览**：
- v0.1：服务器端基础闭环（Docker 硬闭环，deb 可选）
- v0.2：服务器端扩展（rpm + 多语言）
- v0.3：移动端 + Web→移动端（并行开发）
- v0.4：移动端验收
- v0.5：桌面端（可选）
- v1.0：多端完整验收

### 8.2 MVP 用户故事

```text
作为一个学生或社团开发者，
当我有一个 Python 项目需要交付给别人运行时，
我希望可以让 AI Agent 调用 ForgeKit，
生成可运行的 Docker 镜像（deb 包按需可选），
这样我不用先成为打包专家，也能得到可验证的交付产物。
```

**多端用户故事（v0.3+）**：

```text
作为一个移动端开发者，
当我有一个 Android/iOS 项目需要交付给别人安装时，
我希望可以让 AI Agent 调用 ForgeKit，
生成 APK/IPA 包（不上架），
这样我不用处理复杂的证书签名和上架流程，也能直接分发安装包。

作为一个 Web开发者，
当我有一个 Web 项目需要让别人在手机上使用时，
我希望可以让 AI Agent 调用 ForgeKit，
生成 PWA 或混合应用包，
这样我不用重写原生代码，也能提供移动端体验。
```

### 8.3 MVP 必做（v0.1服务器端）

| 模块 | 要求 |
|------|------|
| Agent 接入 | MCP Server 能暴露工具列表和结构化输入输出 |
| 打包计划文档 | 能生成 `Forge.md`，记录目标、策略、风险和结果 |
| Docker 构建 | **硬闭环** | 支持 Python 项目的本地 Docker 镜像构建（linux/amd64） |
| deb 构建 | **可选** | 仅当目标确为 Ubuntu + systemd 时支持 Ubuntu 22.04 x86_64 基础 deb 构建；否则后置 |
| 决策依据 | 每次构建返回目标系统、架构、基础镜像、风险说明 |
| 示例项目 | 提供一个可复现 Python fixture |
| 验收 | 一次手动 E2E：Agent 指令到产物验证 |

### 8.4 MVP 不做（明确不做）

| 不做 | 原因 |
|------|------|
| Web UI | 第一用户是 Agent，不是浏览器用户 |
| 多语言模板（v0.1） | 先验证单语言闭环，v0.2才扩展 |
| rpm | v0.2才扩展，避免 v0.1 同时处理两种包格式 |
| 移动端（APK/IPA） | v0.3才开始移动端，先验证服务器端闭环 |
| Web→移动端（PWA/Hybrid） | v0.3与移动端并行开发 |
| 桌面端（exe/app） | v0.5可选扩展，仅在确认需求后实现 |
| 云构建平台 | 本地优先，降低部署复杂度 |
| 上架流程（Google Play/App Store） | **明确不做**，只做本地打包和分发，上架流程复杂度远超打包 |
| 自动发布到 apt/yum 源 | v0.1 先验证产物生成 |

## 9. 成功指标

| 指标 | v0.1 目标 | v0.2 目标 |
|------|-----------|-----------|
| 真实项目试点 | 3 个 | 10 个 |
| Agent 一次成功率 | 60% | 80% |
| 从源码到产物耗时 | 30 分钟内 | 15 分钟内 |
| 失败可解释率 | 80% | 90% |
| 文档独立可用 | 1 个外部用户可照文档跑通 | 5 个外部用户可跑通 |
| 打包文档可审查 | 3 个试点项目能生成可读 `Forge.md` | 10 个真实项目复用该格式 |

## 10. 风险与应对

| 风险 | 应对 |
|------|------|
| 用户只需要 Docker，不需要 deb | 已拍板 Docker 硬闭环、deb 可选；若试点证伪 deb 需求则后置 v0.2 |
| Agent 调用不稳定 | 工具 Schema 收窄，输出结构化错误和决策依据 |
| 打包范围过大 | 严格限制 v0.1：Python + Ubuntu 22.04 + x86_64 |
| 与现有工具重复 | 不重写底层工具，重点做 Agent 编排、知识解释和低门槛路径 |
| 商业价值不明确 | 先做社团/独立开发者试点，再判断是否扩展为通用产品 |
| 自研协议过早 | v0.1 只支持 MCP 和 Markdown 文档，不发明新协议 |

## 11. 进入开发的前置条件

深度开发前必须完成：

1. 本需求文档通过评审。
2. 设计文档明确架构边界。
3. 技术计划明确目录、接口、测试和开发顺序。
4. 至少完成初步竞品分析和 3 个试点项目候选。

未完成以上条件，不进入能力层开发。
