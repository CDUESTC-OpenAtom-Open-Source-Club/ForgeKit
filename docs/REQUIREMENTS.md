# ForgeKit Requirements

> 版本：v0.1 规划稿
> 阶段：需求定义与调研规划
> 原则：需求、设计、技术计划通过后，才进入深度开发。

## 1. 项目一句话定义（重新定位）

ForgeKit 是 **面向 AI Agent 的可验证交付契约层（Verifiable Delivery-Contract Layer for AI Agents）**。

它不重写打包工具，而是为 AI Agent（以及非专家开发者）提供一套**可决策、可解释、可验证的交付契约**：把"能在本地跑起来的代码"可靠地推进到"可直接分发的多端产物"。核心资产不是打包壳，而是 **Forge.md 决策契约 + 跨端决策知识（decision-rules / 兼容性矩阵）**。

第一版（v0.x）只做**本地优先的交付层（L0 获客层）**：验证 Agent 能否可靠调用一组本地工具完成多端打包，并产出决策依据与可验证结果。商业化所需的云端构建/签名/分发（L2）与标准效应（L3）是后续阶段，不在 v1 范围。

> 验证问题（v0.1 范围）：AI Agent 能否可靠调用一组本地工具，把一个真实项目打成可直接分发使用的多端安装包，并解释"为什么这么构建"。

### 1.1 市场调研结论（为什么是现在，为什么是 ForgeKit）

完整报告见 [`docs/MARKET_RESEARCH.md`](./MARKET_RESEARCH.md)。三方向调研交叉验证了本次转型：

- **需求真实且致命（方向一）**：多步交付任务复利错误率极高（20 步 × 95% → 整体仅 35.8%）；主流 Agent 在打包/部署上多为"沙箱 Shell 硬试"，无交付契约，失败即静默。**63%** 企业领导要求校验所有 Agent 输出。
- **产品形态与蓝海空白（方向二）**：MCP 已是事实标准（300+ Server，全主流 Agent 原生支持），但**没有任何一个"交付契约层" Server**；现有 Server 都是 API 转发器，缺"决策 + 观测"。ForgeKit 的 `Forge.md` + `decision-rules` + `result.json` 恰好填补，应做成标准 MCP Server。
- **商业化护城河（方向三）**：EU AI Act 2026-08 强制执行、82% 企业已有 AI 治理顾虑、Docker MCP 目录强制 SBOM + provenance——可解释、可审计的交付链是企业付费点（对应 L2/L3）。

**研判**：牵头方向为方向二；v0.1 在"最便宜表面（服务器端 Docker）"上证明契约成立，是单位投入战略价值最高的一版（见 §8 每一届 MVP 阶梯）。

**关键区别（vs 传统打包工具）**：
- ✅ 面向 Agent 的契约接口（MCP + Skill + Forge.md），而非只给人用的专家 CLI
- ✅ 输出**决策依据与可行性矩阵**，而不只是命令
- ✅ 跨端决策知识沉淀为可检索、可机器调用的结构化数据
- ✅ 本地优先、不上架；云端构建/分发作为后续商业化层

## 2. 目标用户（需求从哪来）

ForgeKit 的需求不是"用户不会打包"，而是"**AI Agent 成为主流后，交付环节是断点**"——Agent 能写代码，却没有可靠、可解释的 shipping 接口；非专家也卡在"最后一公里"。

| 用户 | 典型场景 | 当前困难 | ForgeKit 价值 |
|------|----------|----------|---------------|
| 学生/社团开发者 | 课程/社团项目需多端交付 | 会写代码，但不会 Dockerfile、Gradle、签名 | 用 Agent 指令完成多端交付闭环 |
| 独立开发者 | 小工具要交付 Docker/deb/rpm/APK/IPA/exe | 不想维护复杂构建配置与证书 | 降低多端交付维护成本 |
| Web 开发者 | Web 项目要变成移动端可用版本 | 不懂如何把 Web 适配到移动端 | Web→移动端一键交付 |
| AI Agent 用户 | 已用 Claude Code/Cline/Cursor 写码 | Agent 会改代码，但缺可靠交付工具契约 | 给 Agent 标准化交付接口与决策依据 |
| AI Agent 平台/集成商 | 想给自家 Agent 加"发布/交付"能力 | 自研 shipping 成本高、跨端知识难攒 | 把 ForgeKit 当作默认交付后端接入 |

## 3. Agent 接入与使用方式

ForgeKit 不是只解决"所有平台打包"的问题，更重要的是解决"Agent 如何可靠使用打包能力"的问题。

v0.1 的接入策略是：

| 接入方式 | 定位 | v0.1 决策 |
|----------|------|-----------|
| MCP Server | 主协议，给 Codex、Claude Code、Cline、Cursor 等 Agent 调用工具 | 必做 |
| Agent Skill / Markdown Guide | 辅助说明，告诉 Agent 如何规划、确认、调用和总结 | 必做文档，不先做运行时 | |
| `Forge.md` | 项目级打包计划文件，记录目标、策略、风险和结果 | 必做 |
| 自研协议 | 未来可能存在的高级打包语义协议 | v1.0 前不做 |

使用流程：

1. Agent 读取项目。
2. Agent 生成或更新 `Forge.md`。
3. 用户确认打包目标。
4. Agent 调用 ForgeKit MCP 工具执行构建。
5. ForgeKit 将结果和风险写回打包文档。

详细协议规划见 [AGENT_INTEGRATION.md](./AGENT_INTEGRATION.md)，打包文档规范见 [PACKAGING_DOCUMENT.md](./specs/PACKAGING_DOCUMENT.md)；项目框架规范与适配见 [PROJECT_FRAMEWORK.md](./specs/PROJECT_FRAMEWORK.md)。

## 4. 需求来源（用户为什么需要这玩意儿）

很多初中级开发者的项目失败点不在代码本身，而在"最后一公里"；而 AI 编码 Agent 成为主流后，**交付环节成了新的断点**：

- 开发者不知道项目该交付 Docker 镜像、系统包还是安装包；不清楚目标系统/架构/版本如何影响产物。
- 构建失败后，不知道该改代码、改配置、改依赖还是改发布目标。
- **AI Agent 能写代码，却没有可靠、可解释的"交付接口"**——它不知道该交付什么形态、选哪个版本、怎么签名、失败了改哪里。
- 现有专业工具（Docker/Gradle/Xcode）能力强但是"专家工具"，不提供决策依据，也不对 Agent 友好；跨端决策知识碎片化，缺乏可机器调用的统一层。

ForgeKit 要回答的核心问题是：

> 如何把发布工程中的重复知识、系统差异和构建步骤，整理成 **Agent 可调用、用户可理解、结果可验证** 的交付与验证层？

## 5. 替代方案与缺口

| 方案 | 能解决什么 | 缺口（ForgeKit 的切入点） |
|------|------------|------|
| GitHub Actions / GitLab CI | 自动化流水线 | 新手要写 YAML/Secrets/矩阵；无 Agent 交付契约 |
| Dockerfile / Buildpacks | 容器化交付 | 不解决多端（deb/rpm/APK/IPA/exe）与决策解释 |
| fpm / nfpm / GoReleaser | 快速生成包 | 专家 CLI，需理解格式依赖，无 Agent 接口 |
| EAS / Appflow / Codemagic | 移动端托管构建 | 单端深耕，跨端决策知识与 Agent 契约缺失 |
| Claude Code / Cline / Cursor | 代码生成修改 | **缺稳定、可验证、面向交付工程的工具契约** |

ForgeKit 的差异点**不是"重新发明打包工具"**，而是填补"Agent 交付层"空白：

1. 面向 AI Agent 暴露清晰、可验证的交付契约（MCP + Skill + Forge.md）。
2. 把跨端打包知识、系统适配、错误诊断沉淀为**可检索、可机器调用的结构化数据**。
3. 输出决策依据（可行性矩阵/门槛/安全锚点），而非只输出命令。
4. 本地优先做获客层（L0）；云端构建/签名/分发作为后续商业化层（L2），不提前铺云。

## 6. 商业价值判断与护城河（分层）

ForgeKit 的价值必须沿栈爬升，不能停在"交付层功能"——否则只是大平台 Agent 的一个内置 feature，会被吞掉。价值分层如下：

| 层 | 定位 | 价值/营收方式 | 护城河 |
|----|------|--------------|--------|
| L0 交付层（免费/开源） | 本地 Agent 交付契约 + Forge.md 规范 | 获客、建生态、成标准入口 | Agent 采用率 |
| L1 知识数据资产 | decision-rules + 兼容性矩阵（跨端策展知识） | 付费规则包、兼容更新、企业定制 | 数据随真实构建反馈增值（观测回溯闭环） |
| L2 构建执行/供应链 | 云端构建、签名金库、产物仓库、可验证链路 | 订阅/按量 recurring revenue | 锁定 + 真实构建数据 |
| L3 标准/平台效应 | Forge.md 成为"如何交付任意项目"的通用清单标准 | 战略价值、被收购门票 | 标准网络效应 |

**护城河来源**：① 数据闭环——每次构建的成败/state_delta 回流知识库，越用越准（巨头缺跨端策展知识）；② 标准——若 Forge.md 成交付清单事实标准，价值指数级放大；③ Agent 采用率——被越多 Agent/平台接入，切换成本越高。

**诚实风险**：只做 L0 是 feature 不是 business；必须主动爬到 L2（云端构建/分发）与 L3（标准）。v0.x 刻意本地优先只是 L0 获客策略，不意味着商业终点。

**市场调研对商业化的进一步支撑（方向三）**： EU AI Act 于 2026-08-02 全面强制执行；**82%** 企业已有 AI 治理顾虑；Docker MCP 目录已**强制 SBOM + provenance** 才能上架。这些信号共同证明"可解释、可审计、安全可控的 AI 交付链"是企业真实付费点，为 L2（云端构建/签名/分发）与 L3（标准效应）提供了叙事与营收基础。

短期目标（v0.x）仍是**验证需求而非收费**：

- 是否有人愿意把真实项目交给 ForgeKit 交付。
- 是否明显减少"代码可运行 → 产物可交付"的时间。
- 是否提升 Agent 完成交付任务的一次成功率。

## 7. 需求调研计划

进入深度开发前，需要完成最低限度需求验证。

### 6.1 访谈对象

| 类型 | 数量 | 要求 |
|------|------|------|
| 社团/学生开发者 | 5 人 | 至少有一个需要部署或分享的项目 |
| 独立开发者 | 3 人 | 有 Docker、Release 或安装包经验 |
| AI Agent 用户 | 2 人 | 使用过 Claude Code/Cline/Cursor 做项目 |

### 6.2 访谈问题

1. 你最近一次需要交付项目是什么时候？最后交付了什么？
2. 你现在如何构建 Docker 镜像、安装包或 Release？
3. 哪一步最容易失败或最耗时间？
4. 你是否需要 deb/rpm，还是 Docker 已经足够？
5. 如果 AI Agent 帮你发布项目，你最担心什么？
7. 你能接受项目里新增 `Forge.md` 作为打包计划吗？
8. 你愿意提供一个真实项目做 v0.1 验证吗？

### 6.3 通过标准

| 验证项 | 最低标准 |
|--------|----------|
| 真实痛点 | 10 个访谈对象中至少 5 个最近一年遇到发布/打包问题 |
| 试点项目 | 至少 3 个真实项目愿意参与 v0.1 验证 |
| MVP 接受度 | 至少 3 个用户接受 v0.1 只做 Python + Docker/deb 的窄范围 |
| 替代方案缺口 | 能明确说出现有工具为什么对目标用户不够友好 |

**已拍板（v0.0 退出）**：v0.1 采用「Docker 硬闭环 + deb 可选」。若后续试点证明用户只需 Docker，deb 直接后置到 v0.2，不做硬验收。

## 8. 每一届 MVP 定义（价值阶梯：v0.1 最高）

> 每届 MVP 不是"加平台"，而是**把同一份可验证交付契约逐端落地**。v0.1 在服务器端证明契约成立，战略价值最高；后续版本扩面不换核。完整路线图见 [ROADMAP.md](./ROADMAP.md)。

| 版本 | 主题 | 核心交付（契约视角） | 战略价值 |
|------|------|----------------------|----------|
| **v0.1** | 契约闭环验证 | 服务器端 Docker 硬闭环 + `Forge.md` / `decision-rules` / `result.json` 三件套强制串联 | ★ 最高：证明论点本身 |
| v0.2 | 契约可靠性 | 错误诊断知识库、多语言（Go/TS）、rpm、GitHub Releases 输出 | 提升一次成功率（攻 50% 困局） |
| v0.3 | 契约跨端化 | 移动端 APK/IPA + Web→移动端 PWA/Hybrid，契约 Schema 跨端复用 | 验证可组合 / 跨端假设 |
| v0.4 | 移动端验收 | 移动端契约稳定验收 | 端到端可用 |
| v0.5 | 桌面端（可选） | Windows exe / macOS app，仅确认需求后做 | 补齐场景 |
| v1.0 | 标准效应 + L2 起步 | 稳定 API、贡献指南、公开案例、起步云构建/签名/分发 | 商业化与标准护城河 |

### 8.1 v0.1 MVP 目标

验证服务器端最小闭环（多端路线的第一阶段）：

> 用户通过 AI Agent 说明打包目标，Agent 调用 ForgeKit，本地生成可验证的服务器端产物（Docker 镜像；deb 按需可选），并解释为什么这么构建。

**多端路线总览**：
- v0.1：服务器端基础闭环（Docker 硬闭环，deb 可选）
- v0.2：服务器端扩展（rpm + 多语言）
- v0.3：移动端 + Web→移动端（并行开发）
- v0.4：移动端验收
- v0.5：桌面端（可选）
- v1.0：多端完整验收

### 8.2 MVP 用户故事

```text
作为一个学生或社团开发者，
当我有一个 Python 项目需要交付给别人运行时，
我希望可以让 AI Agent 调用 ForgeKit，
生成可运行的 Docker 镜像（deb 包按需可选），
这样我不用先成为打包专家，也能得到可验证的交付产物。
```

**多端用户故事（v0.3+）**：

```text
作为一个移动端开发者，
当我有一个 Android/iOS 项目需要交付给别人安装时，
我希望可以让 AI Agent 调用 ForgeKit，
生成 APK/IPA 包（不上架），
这样我不用处理复杂的证书签名和上架流程，也能直接分发安装包。

作为一个 Web开发者，
当我有一个 Web 项目需要让别人在手机上使用时，
我希望可以让 AI Agent 调用 ForgeKit，
生成 PWA 或混合应用包，
这样我不用重写原生代码，也能提供移动端体验。
```

### 8.3 MVP 必做（v0.1服务器端）

| 模块 | 要求 |
|------|------|
| Agent 接入 | MCP Server 能暴露工具列表和结构化输入输出 |
| 打包计划文档 | 能生成 `Forge.md`，记录目标、策略、风险和结果 |
| Docker 构建 | **硬闭环** | 支持 Python 项目的本地 Docker 镜像构建（linux/amd64） |
| deb 构建 | **可选** | 仅当目标确为 Ubuntu + systemd 时支持 Ubuntu 22.04 x86_64 基础 deb 构建；否则后置 |
| 决策依据 | 每次构建返回目标系统、架构、基础镜像、风险说明 |
| 示例项目 | 提供一个可复现 Python fixture |
| 验收 | 一次手动 E2E：Agent 指令到产物验证 |

### 8.4 MVP 不做（明确不做）

| 不做 | 原因 |
|------|------|
| Web UI | 第一用户是 Agent，不是浏览器用户 |
| 多语言模板（v0.1） | 先验证单语言闭环，v0.2才扩展 |
| rpm | v0.2才扩展，避免 v0.1 同时处理两种包格式 |
| 移动端（APK/IPA） | v0.3才开始移动端，先验证服务器端闭环 |
| Web→移动端（PWA/Hybrid） | v0.3与移动端并行开发 |
| 桌面端（exe/app） | v0.5可选扩展，仅在确认需求后实现 |
| 云构建平台 | 本地优先，降低部署复杂度 |
| 上架流程（Google Play/App Store） | **明确不做**，只做本地打包和分发，上架流程复杂度远超打包 |
| 自动发布到 apt/yum 源 | v0.1 先验证产物生成 |

## 9. 成功指标

| 指标 | v0.1 目标 | v0.2 目标 |
|------|-----------|-----------|
| 真实项目试点 | 3 个 | 10 个 |
| Agent 一次成功率 | 60% | 80% |
| 从源码到产物耗时 | 30 分钟内 | 15 分钟内 |
| 失败可解释率 | 80% | 90% |
| 文档独立可用 | 1 个外部用户可照文档跑通 | 5 个外部用户可跑通 |
| 打包文档可审查 | 3 个试点项目能生成可读 `Forge.md` | 10 个真实项目复用该格式 |

## 10. 风险与应对

| 风险 | 应对 |
|------|------|
| 用户只需要 Docker，不需要 deb | 已拍板 Docker 硬闭环、deb 可选；若试点证伪 deb 需求则后置 v0.2 |
| Agent 调用不稳定 | 工具 Schema 收窄，输出结构化错误和决策依据 |
| 打包范围过大 | 严格限制 v0.1：Python + Ubuntu 22.04 + x86_64 |
| 与现有工具重复 | 不重写底层工具，重点做 Agent 编排、知识解释和低门槛路径 |
| 商业价值不明确 | 先做社团/独立开发者试点，再判断是否扩展为通用产品 |
| 自研协议过早 | v0.1 只支持 MCP 和 Markdown 文档，不发明新协议 |

## 11. 进入开发的前置条件

深度开发前必须完成：

1. 本需求文档通过评审。
2. 设计文档明确架构边界。
3. 技术计划明确目录、接口、测试和开发顺序。
4. 至少完成初步竞品分析和 3 个试点项目候选。

未完成以上条件，不进入能力层开发。

## 12. 蓝海扩展方向（双轮驱动：教育价值 + 商业价值）

ForgeKit 的杀手锏是**"抹平环境差异，提供确定性的交付结果"**。在三个主调研方向（见 §1.1 / [`MARKET_RESEARCH.md`](./MARKET_RESEARCH.md)）之外，还存在三片蓝海，它们都由 v0.1 的"可验证交付契约层"支撑，是双轮驱动（教育传播 + 商业变现）的延展：

- **蓝海一 · AI 助教式零基础环境 / 构建沙箱**：消灭高校社团 / 机构新人的环境配置挫败，成为 Onboarding 标配；向高校 / 机构私有化部署，按班级 / 账号订阅。依赖 v0.2 错误诊断 + 多语言。
- **蓝海二 · 私有化边缘构建集群**：用内网穿透串联闲置设备（如 ARM 本）做零公网暴露、零云端算力的私有构建，卖"边缘调度"能力。是 L2 构建执行层的分布式形态。
- **蓝海三 · 跨架构开源无人驾驶发布矩阵**：对接开源社区，自动多架构沙箱编译 + Release 产物 + 测试报告，建立行业技术壁垒（L3 标准效应 / 融资叙事）。

> 三片蓝海都不是 v0.1 范围，但 **v0.1 是它们的共同地基**：先把"确定性交付契约"做硬，蓝海才能站稳。完整分析见 [`docs/BLUE_OCEAN.md`](./BLUE_OCEAN.md)；v0.1 执行路径见 [`docs/V0.1_IMPLEMENTATION.md`](./archive/v0.1-mvp/V0.1_IMPLEMENTATION.md)。
