# ForgeKit 版本范围契约

> 生效日期：2026-07-21
> 作用：作为版本承诺、开发排期和发布验收的统一边界。

## 1. 状态定义

每项能力只能使用以下状态：

| 状态 | 含义 | 可否写入 README 已支持能力 |
|------|------|----------------------------|
| 已实现 | 已有代码，但可能只经过单元测试 | 否 |
| 已验证 | 已在自动化或明确的真实环境完成端到端验证 | 可以，必须写明支持矩阵 |
| 试验性 | 可供试点使用，Schema 或行为可能调整 | 可以，必须标注试验性 |
| 候选 | 尚未排期，等待需求和验收环境 | 否 |

完成代码不等于版本完成。版本只有满足全部退出门槛后才能发布。

## 2. v0.1：可运行核心

### 用户承诺

用户可以通过 MCP 将支持范围内的服务项目转换为本地 OCI/Docker 镜像，并获得计划、日志、决策依据和结构化结果。

### 范围

- MCP 工具：`inspect_project`、`generate_packaging_plan`、`build_docker_image`、`pack_deb`；
- Python 服务项目作为主验收类型；
- `linux/amd64` 本地镜像；
- 强制 Plan-before-build；
- deb 仅为 Ubuntu/systemd 的试验性能力；
- 本地执行，不上传源码，不自动推送产物。

### 非范围

- arm64、多架构 manifest；
- Registry Push、GitHub Release 自动发布；
- Node/Go 的正式兼容承诺；
- 云构建、账户、审批和签名；
- Windows、macOS、Android、iOS。

### 退出门槛

- `npm run verify` 全部通过；
- 编译后 MCP 进程可以发现并调用工具；
- Python/Docker 至少一次真实构建和容器运行验证；
- 缺少 `Forge.md` 时构建必须被阻断；
- README 不包含未验证承诺。

## 3. v0.2：Docker 构建诊断验证

### 用户承诺

非维护者可以在 10 分钟内接入 ForgeKit，在 Docker 构建前发现问题，在失败后得到可操作解释，并在成功后验证镜像真实可运行。

### 范围

- 第五个 MCP 工具 `preflight_check`；
- 第六个只读 MCP 工具 `diagnose_build_failure`，用于分析已有 Docker/BuildKit 失败日志；
- 源目录、计划、容器运行时、磁盘和 Registry 连通性检查；
- 10+ 常见失败模式诊断；
- Release Manifest 作为辅助证据：Git commit、工具版本、环境、产物大小与 SHA256；
- npm/npx 安装入口及版本锁定示例；
- 环境、路径、Dockerfile、依赖、Registry、架构和权限失败样本；
- 诊断结果包含原始证据、可能原因、置信度、安全建议和验证步骤；
- Python、Node.js/TypeScript、Go 代表项目试点；
- Docker Engine 为主支持运行时，Podman CLI 兼容性保持试验性；
- 国内网络环境的镜像源说明，不自动改写用户系统配置。

### 非范围

- 不承诺任意 Python/Node/Go 项目都能自动生成正确 Dockerfile；
- 不自动安装 Docker、配置镜像源或修改系统服务；
- 不自动 commit、tag、push、发布 Release；
- 不做 arm64 正式支持；
- 不建设 Web 控制台或 SaaS。

### 退出门槛

- npm tarball 在全新临时目录安装并通过 MCP 调用；
- 正式发布后的 `npx` 路径在 Linux/macOS 至少各验证一次；
- Python、TypeScript、Go 各至少一个真实镜像可构建并运行；
- 至少 30 个公开失败样本进入回归集；
- 至少 5 名非维护者完成真实诊断；
- 80% 用户在 10 分钟内完成安装接入；
- 失败分类正确率与建议覆盖率均达到 80%；
- 至少 60% 试点用户认为定位时间缩短；
- 至少 2 名用户愿意再次使用。

## 4. v0.3：常见项目广泛可用

### 用户承诺

常见 Python、Node.js/TypeScript 和 Go 服务可以在 Agent 与 CI 中复用同一交付契约，并输出 amd64/arm64 产物。

### 范围

- 三种语言的版本化模板与兼容性矩阵；
- 用户自带 Dockerfile 与 ForgeKit 模板两条明确路径；
- `linux/amd64`、`linux/arm64`；
- GitHub Actions 和至少一种通用非交互 CI；
- OCI 镜像、校验文件和可选 GitHub Release 归档；
- 规则 Schema 版本与迁移策略；
- deb/rpm 仅在真实试点证明需要时扩展。

### 非范围

- 不承诺跨架构构建等同于目标机器运行验证；
- 不自动托管 Registry 凭据；
- 不进入移动端和桌面端全面支持；
- 不提供企业审批和长期审计服务。

### 退出门槛

- 30 个真实项目进入兼容性矩阵；
- 三种语言各至少 5 个项目；
- amd64/arm64 均有构建和运行验收；
- 至少 2 个 MCP 客户端和 1 个 CI 环境持续验证；
- 模板变更必须经过真实容器构建测试，而非只做识别测试。

## 5. v0.4：可信发布

### 用户承诺

团队可以证明产物来自哪个源码和计划、经过哪些策略检查，并获得机器可读的供应链证据。

### 范围

- SBOM；
- provenance/attestation；
- Release Manifest Schema 稳定化；
- 基础组织策略，例如禁止 `latest`、限制基础镜像；
- 人类报告与机器 JSON；
- 可复现性差异说明；
- 发布前策略阻断。

### 非范围

- 不承诺已达到任何特定监管认证；
- 不默认托管私钥；
- 不自动审批或替代安全团队；
- 不建设完整企业 IDP。

### 退出门槛

- 正式产物 100% 有 Manifest、SBOM 和 provenance；
- 关键策略可在构建前阻断；
- 3 个小团队实际使用报告进行发布审查；
- 至少 3 个团队提出共同的策略、审计或托管需求。

## 6. v1.0：稳定开源产品

### 用户承诺

ForgeKit 具有稳定安装入口、稳定 MCP Schema、明确支持矩阵和可扩展适配器边界，适合外部项目持续依赖。

### 范围

- MCP 工具兼容政策；
- `Forge.md`、规则、结果和 Manifest 的版本化规范；
- 语言/产物适配器接口；
- 安全政策、贡献指南、发布节奏和弃用流程；
- 公开案例与长期回归矩阵；
- 遥测仅选择加入，字段透明。

### 非范围

- “支持所有语言、平台和发布目标”；
- 无限制的向后兼容；
- 自动上架 App Store/Google Play；
- 托管云能力作为开源 v1.0 的发布前提。

### 退出门槛

- 至少两个小版本周期保持 Schema 兼容；
- 外部用户和贡献者持续使用；
- 支持矩阵中的组合全部有自动验收；
- 安装、升级、降级和迁移文档经过外部验证；
- 已知限制与安全响应流程公开。

## 7. v1.x：社区与平台扩展

公共候选包括课程案例、社区规则、兼容矩阵、隔离构建教学、SBOM/provenance 实践和开源项目发布辅导。

任何公共能力需要至少 3 个课程、社团或开源团队提出共同问题并愿意参与维护。任何 Windows、macOS、Android、iOS 或 PWA 扩展需要同时具备真实项目、试点用户和自动化验收环境。

## 8. 版本变更规则

- 新能力默认进入候选池，不直接加入承诺版本；
- 从候选进入范围必须有用户问题、实现边界和验收方法；
- 从已实现升级为已验证必须留下自动化或真实环境记录；
- 版本发布后新增破坏性 Schema 变更必须进入下一主/次版本；
- 未达到退出门槛时延后版本，不通过降低验收标准宣布完成。
