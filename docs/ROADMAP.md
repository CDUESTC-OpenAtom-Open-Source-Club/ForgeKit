# ForgeKit Roadmap

> 路线版本：2026-07-21
> 核心原则：版本按用户价值和验证证据推进，不按平台数量推进。
> 当前执行范围以 [CURRENT_OBJECTIVES.md](./CURRENT_OBJECTIVES.md) 为准。

## 1. 路线总览

ForgeKit 的发展路径为：

```text
可运行的核心
  → 外部用户可用
  → 常见项目广泛可用
  → 团队可信发布
  → 按需求扩展平台
```

| 版本 | 主题 | 用户获得的价值 | 进入下一阶段的核心证据 |
|------|------|----------------|------------------------|
| v0.1 | 可运行核心 | Agent 能把项目生成 Docker/OCI 产物并解释结果 | 仓库内验证持续通过 |
| v0.2 | 可用性版本 | 非维护者可以独立安装、接入并完成交付 | 10 个真实项目、80% 首次成功率 |
| v0.3 | 广泛使用版本 | 常见服务项目和 CI 工作流共享同一交付契约 | 30 个项目、3 种语言、2 种架构 |
| v0.4 | 可信发布版本 | 每个产物都有源码、环境、规则和校验依据 | SBOM/provenance 与发布证据完整 |
| v1.0 | 稳定开源产品 | 稳定协议、兼容承诺、公开案例和扩展机制 | 外部采用与贡献持续发生 |
| v1.x | 商业与平台扩展 | 团队治理、托管执行及需求驱动的多端能力 | 明确的付费试点或平台需求 |

## 2. v0.1：收敛现有核心

### 目标

把现有四个 MCP 工具收敛为一条真实、可重复的 Python/Docker 主闭环，不扩大产品表面。

### 必须完成

- `inspect_project → generate_packaging_plan → build_docker_image` 全链路通过；
- `pack_deb` 保持可选，并只在 Ubuntu/systemd 场景推荐；
- 构建工具强制检查有效 `plan_path`；
- 结果包含 `decision_basis`、日志路径、产物标识和 SHA256；
- 单元、MCP 协议、编译后运行和至少一次真实 Docker E2E 验证；
- README 只陈述已经验证的能力。

### 发布门槛

- `npm run verify` 持续通过；
- 全新环境可按文档启动 MCP Server；
- 至少一个维护者之外的用户完成示例项目交付；
- 所有构建结果都能区分成功、失败和未执行。

## 3. v0.2：可用性版本

### 目标

把“开发者能运行”提升为“目标用户无需理解内部架构也能使用”。

### 产品工作

- 提供稳定的 npm/npx 安装入口与版本锁定方式；
- 提供 Claude Code、Codex、Cursor、Cline 等客户端的最短配置示例；
- 增加 preflight/dry-run，提前检查工具链、权限、磁盘和项目入口；
- 为常见失败返回错误类别、原因、建议动作和重试条件；
- 提供 `doctor` 或等价诊断能力，但优先复用现有 MCP 接口，避免重复协议；
- 将 Git commit、dirty state、ForgeKit 版本、目标和产物哈希写入 Release Manifest；
- 建立脱敏规则，默认不上传源码、密钥和完整环境变量；
- 完成 10 个真实项目试点及公开案例。

### 发布门槛

- 80% 新用户 10 分钟内完成安装接入；
- 支持范围内首次交付成功率 ≥ 80%；
- 失败可解释率 ≥ 90%；
- 至少 5 名外部用户仅按文档完成交付；
- 100% 构建关联可追溯的源码和工具版本。

## 4. v0.3：广泛使用版本

### 目标

扩展常见项目和工作流覆盖，而不是盲目增加终端平台。

### 扩展顺序

1. Python、Node.js、Go 服务型项目；
2. 用户已有 Dockerfile 与 ForgeKit 推荐模板两种路径；
3. linux/amd64 与 linux/arm64；
4. GitHub Actions 和通用 CI 非交互模式；
5. OCI 镜像、校验文件和 GitHub Release 归档；
6. deb/rpm 只根据试点需求继续扩展。

### 发布门槛

- 至少 30 个真实项目进入兼容性矩阵；
- 三种语言各有不少于 5 个真实项目；
- amd64/arm64 都有自动化验收；
- 至少两个 MCP 客户端和一个 CI 环境持续验证；
- 规则变更具有 Schema 版本、迁移说明和回归样本。

## 5. v0.4：可信发布版本

### 目标

让 ForgeKit 的价值从“方便打包”上升到“能够证明这个产物从哪里来、为什么可信”。

### 核心能力

- 生成 SBOM；
- 生成或对接 provenance/attestation；
- 对构建计划、规则版本、源码提交和产物建立不可歧义关联；
- 支持组织级策略检查，例如禁止 latest 标签、禁止高危基础镜像；
- 提供人类可读报告和机器可读 JSON；
- 明确可复现构建的支持边界，并报告环境差异。

### 发布门槛

- 所有正式产物都有完整 Release Manifest；
- 关键策略违反能够在构建前阻断；
- 至少 3 个小团队将报告用于发布审查；
- 至少 3 个团队表达对策略、审计或托管能力的付费试用意愿。

## 6. v1.0：稳定开源产品

### 目标

形成一个可安装、可集成、可扩展、具有兼容承诺的 Agent 交付契约产品。

### 必须具备

- 稳定的 MCP 工具 Schema 和兼容策略；
- `Forge.md`、规则和结果格式的版本化规范；
- 插件/适配器边界，允许社区增加语言和产物类型；
- 清晰的支持矩阵，不用模糊措辞承诺“全平台”；
- 公开案例、贡献指南、安全政策和发布节奏；
- 匿名遥测必须选择加入，且数据字段透明可审查。

## 7. v1.x：商业化与需求驱动的平台扩展

### 商业产品候选

- ForgeKit Team：私有规则、审批流、角色权限和审计保留；
- ForgeKit Cloud：隔离构建、缓存、产物仓库和按量执行；
- ForgeKit Trust：签名密钥、SBOM、provenance 和政策证明；
- 企业私有部署、长期支持和兼容认证。

### 平台扩展门禁

Windows、macOS、Android、iOS、PWA 或特定 Linux 发行版，只有满足以下条件才进入排期：

1. 至少 5 个真实项目提出同类需求；
2. 至少 2 个用户愿意参加试点；
3. 已有可自动执行的构建与安装验收环境；
4. 不要求重写现有交付契约；
5. 能明确说明该能力带来的用户价值或收入路径。

## 8. 版本控制与项目发布融合

ForgeKit 不接管 Git 操作，而是消费和记录 Git 状态。

每次交付至少记录：

- repository（可脱敏或仅保留本地标识）；
- commit SHA、branch/tag 和 dirty state；
- `Forge.md` 与规则 Schema 版本；
- ForgeKit 版本和构建目标；
- 构建环境摘要；
- 产物名称、类型、大小和 SHA256；
- 执行状态、错误类别和时间戳。

推荐发布流程为：

```text
选择 Git commit
  → 生成/审查 Forge.md
  → preflight
  → 构建与验证
  → 生成 Release Manifest
  → 用户确认
  → 由 GitHub/GitLab/Registry 发布
```

Tag、Push、GitHub Release 和 Registry Push 都是外部状态变更，必须由用户明确启用；默认仅生成本地产物和发布证据。

## 9. 路线决策规则

任何新增功能进入路线图前都必须回答：

1. 哪类真实用户现在无法完成什么任务？
2. 现有四个工具或契约为什么不能较小扩展解决？
3. 如何自动验证它真的可用？
4. 会增加多少安装、维护和支持成本？
5. 对使用量、成功率、可信度或收入的哪一项产生可测改善？

如果这些问题没有证据，功能进入候选池，而不是承诺版本。
