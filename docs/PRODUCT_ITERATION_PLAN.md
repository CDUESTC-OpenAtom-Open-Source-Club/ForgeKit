# ForgeKit 产品迭代计划

> 状态日期：2026-07-22  
> 当前版本线：v0.2 Docker 构建诊断验证  
> 决策依据：[公开问卷证据综述](./validation/QUESTIONNAIRE_SYNTHESIS_2026-07-22.md)

## 1. 迭代目标

v0.2 不以功能数量为完成标准，只验证一个结果：

> 用户能否把一次真实 Docker 构建失败交给 ForgeKit，并比直接阅读原始日志更快得到正确、可执行且能够验证的下一步。

版本推进顺序固定为：

```text
v0.2.1 诊断可信
  → v0.2.2 接入顺畅
  → v0.2.3 真实试点
  → 继续、收缩或转向决策
```

上一小版本未达到退出门槛时，不用新增平台或产物掩盖问题。

### 本轮开发前的技术差距

现有 `ErrorDiagnostician` 已能识别 Docker daemon、COPY、npm/pip、权限、端口和网络等少量模式，并接入 `build_docker_image`。但它仍是早期规则集合：部分规则依赖单行文本，输出没有置信度和独立验证步骤，建议中存在无条件 `sudo`、放宽权限或绕过依赖检查的风险，也不能直接分析用户在 ForgeKit 之外产生的 BuildKit 日志。

### 当前实现状态

第一轮代码已完成统一结构、只读 MCP 入口、构建失败复用、输入与路径边界、日志脱敏、16 条最小复现样本和 10 条带来源的公开开发样本。统一评估器已经输出准确率、未知率和混淆结果。开发样本参与过规则修正，不能据此声称已达到真实场景 80% 准确率。v0.2.1 剩余工作是继续扩充可追溯日志、建立人工标注答案，并用未参与规则开发的留出集盲测。

## 2. v0.2.1：诊断内核

### 用户结果

用户提供已有 Docker/BuildKit 失败日志后，得到结构化诊断；使用 ForgeKit 构建失败时也获得相同结构。

### 必须交付

- 建立脱敏的失败样本格式和回归目录，首批覆盖至少 15 个真实或最小复现样本；
- 将诊断输出统一为 `category`、`summary`、`evidence`、`probable_cause`、`confidence`、`suggested_actions`、`verification`；
- 同时分析 stdout 与 stderr，支持多行 BuildKit、npm、pip 和系统错误；
- 优先覆盖路径/COPY、Docker daemon、Dockerfile 语法、npm/pip 依赖、Registry/网络、权限、磁盘空间；
- 每条建议注明适用条件，禁止把 `sudo`、放宽权限或绕过依赖检查作为无条件默认方案；
- 诊断未知错误时保留关键证据并诚实返回 `unknown`，不生成猜测性修复；
- 增加只读 MCP 工具 `diagnose_build_failure`：接收互斥的 `log_text` 或 `log_path`，可选 `source_dir` 用于补充上下文；限制输入大小，不执行修复命令、不修改项目；
- `build_docker_image` 和 `diagnose_build_failure` 复用同一个内部诊断 API，避免两套分类结果。

### 退出门槛

- 至少 15 个样本全部进入自动回归；
- Top 类别分类正确率不低于 80%；
- 每个已识别结果都有原始证据和验证步骤；
- 错误日志脱敏测试覆盖 Token、常见密钥、用户名和绝对路径；
- 未识别错误不会返回高置信度建议。

## 3. v0.2.2：安装与配置

### 用户结果

非维护者能在 10 分钟内安装 ForgeKit、生成适合其 MCP 客户端的配置，并确认 Server 可被发现。

### 必须交付

- 验证 npm tarball 和正式 `npx` 两条路径；
- 为 Codex、Claude Code、Cursor、Cline 提供最短配置，明确哪些组合经过实际验证；
- 提供只读的配置展示与校验能力，不默认改写用户配置；
- 对 Node 版本、命令路径、JSON 语法、Server 启动失败给出明确提示；
- 用全新临时目录和至少 Linux/macOS 两类环境做安装验收；
- 记录首次接入耗时和人工介入次数。

### 退出门槛

- 支持矩阵中的配置均有自动或人工验收记录；
- 5 次干净环境演练中至少 4 次在 10 分钟内完成；
- 配置失败能区分 Node、路径、配置语法和进程启动问题；
- 文档不宣称未实际验证的客户端或运行时。

## 4. v0.2.3：真实项目试点

### 用户结果

真实用户使用自己的公开或脱敏项目完成诊断，并留下可比较的时间与结果证据。

### 必须交付

- 将失败样本扩展到至少 30 个；
- 招募至少 5 名可区分的非维护者，优先选择最近 30 天有 Docker 构建失败的人；
- 每次试点记录原始任务、基线定位时间、ForgeKit 定位时间、建议是否采用、最终是否修复；
- 首次试用后一周内追踪是否发生第二次使用；
- 发布脱敏案例，失败案例与成功案例都保留。

### 退出门槛

- 5 名非维护者完成真实诊断；
- 失败分类正确率与可操作建议覆盖率均达到 80%；
- 至少 60% 参与者的定位时间缩短；
- 至少 2 名参与者完成第二次使用；
- 没有因默认写操作、危险命令或日志泄露造成严重事件。

## 5. v0.3 进入决策

只有 v0.2.3 达标后才选择 v0.3 路线：

| 证据结果 | 决策 |
|---|---|
| 诊断显著节省时间，多个语言重复出现同类问题 | 扩展 Python、Node.js、Go 诊断与 CI 集成 |
| 用户只需要静态检查和日志解释 | 收缩为 Dockerfile/BuildKit doctor，不扩展打包平台 |
| 用户主要卡在 MCP 配置 | 先强化 ForgeKit 接入诊断；达到 3 个独立项目共同需求后再评估通用配置工具 |
| 用户需要可审查的自动打包 | 将计划预览、差异展示、构建验证作为可选工作流，不默认发布 |
| 无法获得 5 名试点或没有重复使用 | 冻结产品扩张，保留教学案例和诊断规则成果 |

## 6. 候选池，不进入当前承诺

- Registry Push、GitHub Release 自动发布；
- 通用 MCP 客户端配置管理；
- deb/rpm 正式扩展；
- arm64 正式支持；
- Windows、macOS、Android、iOS、PWA；
- 云构建、账户、遥测和商业化平台。

候选能力必须满足路线图中的真实项目、试点用户和自动验收门禁后才能排期。
