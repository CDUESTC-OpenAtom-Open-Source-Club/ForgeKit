# ForgeKit 当前目标与范围

> 状态日期：2026-07-20  
> 当前阶段：教学使用、现有能力完善、硬性验证  
> 原则：先把已经成立的契约讲清楚、测扎实；多版本和多平台适配只作为后续规划。

## 1. 当前唯一主线

ForgeKit 当前不以快速增加平台数量为目标，而是作为一套可教学、可演示、可复现的 **AI Agent 可验证交付契约**继续完善。

当前工作聚焦四件事：

1. 用 `Forge.md + decision-rules + result.json` 教学和演示 Plan-before-build。
2. 保证现有 Python/Docker 主闭环以及可选 deb 能力可重复验证。
3. 完善 MCP 工具的说明、结构化错误、测试和真实运行验证。
4. 收集教学与试点反馈，再决定是否启动下一阶段扩张。

## 2. 当前已实现能力

当前 MCP Server 暴露四个工具：

| 工具 | 当前用途 |
|------|----------|
| `inspect_project` | 识别项目语言、入口和已有打包配置 |
| `generate_packaging_plan` | 生成可审查的 `Forge.md` |
| `build_docker_image` | 在存在计划后构建 Docker 镜像 |
| `pack_deb` | 面向 Ubuntu/systemd 场景的可选 deb 构建 |

当前可信主闭环为：

```text
Agent 读取项目
  → inspect_project
  → generate_packaging_plan
  → 人审查 Forge.md
  → build_docker_image / pack_deb
  → 结构化结果与决策依据
```

## 3. 当前硬性边界

以下内容不属于当前必须实现或验收范围：

- Windows exe、macOS app、Linux AppImage 实际构建。
- 网页 AI 应用自动转换为小程序。
- 技能集市、上架、账户和权限系统。
- 全版本留存、增量包、版本差异比较。
- 为大量 Linux 发行版逐个维护底层依赖适配。

文档中出现这些内容时，一律视为候选路线，不得当作当前能力对外承诺。

## 4. 后续三阶段候选规划

以下规划用于保存产品方向，不代表已经启动。

### 阶段 1：基础跨平台打包核心

- 输入：Python/Node AI Agent、本地工具脚本。
- 候选输出：Windows exe、macOS app、Linux AppImage。
- 辅助能力：依赖扫描、冗余文件识别、小资源内置与大模型外置策略。
- 边界：不做网页/小程序转换，不做上架。

启动前必须先验证：跨系统构建节点、签名/公证、杀毒误报、AppImage 的 glibc 兼容基线和真实用户需求。

### 阶段 2：统一 AI 技能包

- 把桌面程序、网页能力、小程序组件、函数插件和向量库抽象为独立技能单元。
- 统一描述技能说明、依赖、入口、资源、配置、权限和平台适配器。
- 支持单技能细粒度打包，也支持完整应用合集。
- 支持技能组合与复用。

网页与小程序之间优先复用“可移植 AI 核心 + 平台适配器”，不承诺任意页面无损自动转换。

### 阶段 3：校内/社团分发集市

- 技能名称、平台、简介、教程和版本记录。
- 分类检索、下载、历史版本和一键部署。
- 面向社团/同学的轻量访问控制。
- 最低安全要求：哈希校验、权限声明、发布者追踪和依赖风险提示。

## 5. Linux 后续取舍

AppImage 可减少发行版差异，但不能保证在所有 Linux 上无条件运行。若进入跨平台阶段，应采用较老的 glibc 构建基线并维护一组明确的验证发行版。Ubuntu、openEuler、统信等专属构建只在真实需求成立后增加，小众发行版不主动铺开。

## 6. 启动后续阶段的门槛

只有同时满足以下条件，才把多版本或多平台规划转为执行任务：

1. 当前 `npm run verify` 持续通过。
2. 至少有真实教学或试点反馈证明新平台需求存在。
3. 为新增平台定义了可自动执行的验收环境。
4. 当前契约格式无需因新增平台整体重写。

在门槛满足前，优先完善教学材料、案例、错误解释和测试证据。
