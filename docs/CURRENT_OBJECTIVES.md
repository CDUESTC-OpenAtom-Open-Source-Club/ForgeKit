# ForgeKit 当前目标与范围

> 状态日期：2026-07-21
> 当前阶段：可用性验证（Usability Validation）
> 产品原则：先让真实用户稳定完成一次交付，再扩展平台数量。

## 1. 当前唯一主线

ForgeKit 当前的首要目标，不是成为覆盖所有平台的打包工具箱，而是成为 **AI Agent 可直接调用、普通开发者能够理解、交付结果可以验证的发布入口**。

当前只解决一个完整问题：

> 将一个真实 Git 项目的指定提交，通过可审查计划，转换为可运行、可校验、可追溯的交付物。

标准闭环为：

```text
Git commit / source tree
  → inspect_project
  → generate_packaging_plan
  → 人或策略确认
  → build_docker_image / pack_deb
  → artifact + result.json + release manifest
```

其中：

- Git 管理源码历史、分支和提交；
- `Forge.md` 记录本次交付目标、决策和风险；
- `decision-rules.yaml` 提供可复用的交付约束；
- `result.json` 记录执行结果和失败诊断；
- Release Manifest 将 Git 提交、ForgeKit 版本、构建环境、产物和校验和关联起来。

## 2. 产品优先级

所有开发工作按以下顺序判断，前一项没有达到门槛时，不为后一项大规模扩张：

1. **可安装**：新用户能在 10 分钟内安装并接入一个主流 MCP 客户端。
2. **可完成**：真实项目能从源码走到可运行产物，而不只是生成计划。
3. **可理解**：成功和失败都有清晰、可操作的结构化说明。
4. **可重复**：同一提交和目标可再次构建，并能比较关键差异。
5. **可集成**：可进入 GitHub Actions、GitLab CI 或其他 Agent 工作流。
6. **可扩展**：在上述能力稳定后，再增加语言、产物和平台。

## 3. 当前目标用户

### 核心用户

- 已经使用 Codex、Claude Code、Cursor、Cline 等 Agent 的个人开发者；
- 需要把 Python、Node.js 或 Go 服务交给他人运行的小团队；
- 需要教学或演示“从代码到可信产物”全过程的课程和社团。

### 暂不作为核心用户

- 已有成熟内部开发平台和专业发布团队的大型企业；
- 需要 App Store、Google Play 全自动上架的移动应用团队；
- 要求 ForgeKit 替代 Git、Docker、Gradle、Xcode 或 CI 平台的用户。

当前优先服务个人开发者、学生、教师和开源社团；企业场景不是项目发展的必要终点。

## 4. 当前必须完成的可用产品

当前能力以五个 MCP 工具为基础：

| 工具 | 当前职责 | 可用性要求 |
|------|----------|------------|
| `inspect_project` | 识别项目、入口和现有构建配置 | 对不支持的项目给出明确原因和下一步 |
| `preflight_check` | 构建前检查环境、计划、磁盘与 Registry | 每项明确返回通过、失败或跳过，不修改系统配置 |
| `generate_packaging_plan` | 生成可审查的 `Forge.md` | 默认值合理，减少必须由用户填写的字段 |
| `build_docker_image` | 构建服务器端 OCI/Docker 产物 | 强制计划前置，并验证镜像真实存在 |
| `pack_deb` | 生成 Ubuntu/systemd 场景的可选 deb | 只在明确适用时推荐，不作为普遍默认 |

本阶段必须补齐的不是新平台，而是完整用户体验：

- 一条复制即可使用的安装和 MCP 配置路径；
- 一个无需了解 ForgeKit 内部概念的入门命令或 Agent 指令；
- Python、Node.js、Go 三类代表性项目的真实样例；
- dry-run / preflight 检查，构建前暴露缺少 Docker、入口或权限等问题；
- Git 提交、脏工作区状态、ForgeKit 版本和产物 SHA256 的关联记录；
- 结构化失败分类、建议动作和可分享的脱敏诊断报告；
- 从零开始的公开演示和可重复验收脚本。

每一版本的详细承诺与退出条件见 [VERSION_SCOPE.md](./VERSION_SCOPE.md)。

## 5. 广泛性的定义

ForgeKit 的“广泛性”按以下顺序扩展：

1. **工作流广泛性**：能被多个主流 Agent 和 CI 环境调用；
2. **项目广泛性**：覆盖常见服务型项目，而不是只适配一个示例；
3. **环境广泛性**：支持 amd64/arm64 和明确的运行基线；
4. **产物广泛性**：从 OCI 镜像扩展到 Release 归档、系统包；
5. **平台广泛性**：最后才按真实需求进入桌面、移动端和专属系统。

因此，“增加 APK/IPA/exe”本身不算项目进步；只有外部用户能稳定获得可用产物，才算覆盖扩张。

## 6. 阶段退出门槛

当前阶段只有同时满足以下条件才结束：

| 指标 | 最低门槛 |
|------|----------|
| 外部试点 | 10 个真实项目，至少 3 名非维护者 |
| 安装接入 | 80% 的新用户能在 10 分钟内完成 |
| 首次交付 | 支持范围内一次成功率不低于 80% |
| 失败解释 | 90% 的失败能给出分类和可操作建议 |
| 交付时间 | 典型项目从接入到获得产物不超过 15 分钟（不含依赖下载异常） |
| 可追溯性 | 100% 结果关联源码提交、工具版本、目标和产物校验和 |
| 文档独立性 | 至少 5 名外部用户仅按文档完成闭环 |

未达到门槛时，应修复安装、默认值、诊断、兼容性和文档问题，不用新增平台掩盖基础体验缺口。

## 7. 当前明确不做

- 替代 Git 或创建另一套源码版本控制系统；
- 同时建设 Web 控制台、账户、市场和云构建平台；
- 无真实试点支撑的 Windows、macOS、Android、iOS 全面扩张；
- 自动提交、推送、打 Tag 或发布 Release，除非用户明确确认；
- 默认上传源码、构建日志或环境信息；
- 宣称支持尚未进入自动化验收矩阵的平台。

## 8. 公共开源边界

ForgeKit 作为社团公共开源项目维护，任务是获得真实使用、教学反馈和兼容性数据，不以收费为目标。

未来公共能力建立在开源核心之上：

- 初学者教程、课程案例和工作坊；
- 公共规则库、失败样本与兼容性维护；
- SBOM、provenance 和供应链知识教学；
- 开源项目发布辅导与社区适配器；
- 清晰透明的维护、贡献和版本治理。

只有当可用性门槛满足且真实用户提出共同需求，才扩展新的公共能力；不以付费意愿作为版本启动条件。
