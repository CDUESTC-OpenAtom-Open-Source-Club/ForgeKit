# ForgeKit 市场调研报告

> 版本：v0.1 战略调研稿
> 关联文档：[REQUIREMENTS.md](./REQUIREMENTS.md)（价值与需求）、[DESIGN.md](./DESIGN.md)（框架）、[AGENT_INTEGRATION.md](./AGENT_INTEGRATION.md)（接入形态）
> 调研方法：公开行业报告 + 主流 AI Coding Agent / MCP 生态一手文档交叉验证（2025–2026）
> 转型背景：从"多端打包工具箱"升级为"面向 AI Agent 的可验证交付契约层"

---

## 0. 结论速览（给决策者）

三个调研方向构成**因果链**，而非互斥选项：

```
方向一 需求存在性(已验证)  →  方向二 产品形态 + 蓝海空白(定方向)  →  方向三 商业化护城河(定未来)
   需求侧"为什么做"              供给侧"做成什么 / 怎么寄生"              商业侧"凭什么赚钱"
```

**牵头方向 = 方向二（MCP 协议生态与集成标杆）**：它唯一能拍板产品形态（标准 MCP Server），并暴露出明确蓝海空白——在 300+ MCP Server 中，没有任何一个"交付契约层"。

---

## 1. 方向一：主流 AI Coding Agent 的"最后一公里"交付缺陷

**调研目标**：验证"Agent 缺乏可靠交付契约"是否普遍且致命，并锁定 v0.1 优先交付场景。

### 调研结论：痛点普遍、致命，且主流 Agent 无原生解法

| 证据 | 数据 | 含义 |
|------|------|------|
| 复利错误率 | 20 步 × 95% 单步 → 整体完成率仅 **35.8%**；开源框架真实负载平均完成率 ~**50%**（arXiv:2508.13143） | 多步交付任务天然高失败 |
| 工具调用本身不可靠 | 最佳模型嵌套工具调用仅 **77.5%** 准确（Berkeley 基准） | >20% 概率畸形参数 |
| 可靠性随次数衰减 | 单次 60% 成功 → 连续 8 次跌至 **25%**（arXiv:2511.14136） | 不可用于生产 |
| 企业被迫人工校验 | 要求校验所有 Agent 输出的领导达 **63%**（2025 初仅 22%，Deloitte 2025） | 静默幻觉是头号顾虑 |
| 投产失败率 | GenAI 部署 **70–85%** 未达 ROI；2025 年 **42%** 公司废弃大部分 AI 计划（NTT DATA） | 交付断层是主因 |

**失败集中在终末"交付"四类架构脆弱**：
1. 上下文耗尽（最终步骤被长 JSON 淹没，真正需要的 200 token 埋在中部）；
2. 工具调用级联失败（畸形参数 → 静默跳过或伪造成功）；
3. 规划循环（混淆"重试步骤"与"重订计划"，耗尽上下文与预算）；
4. **后期静默幻觉**（最危险：格式完美、任务未真完成，无报错）。

SWE-bench 失败多位于"跨文件集成 / 最终验证"——即**交付包无法正确组装**。

**主流 Agent 在打包/部署上 = 沙箱内 Shell 硬试，无"交付契约"**：
- **Devin / OpenHands**：自主执行真实工程任务，但交付依赖沙箱内 Shell + 工具调用，无交付契约概念，失败即静默或伪造成功。
- **Cursor Background Agent**：云端 tmux 终端跑命令，文档明示要用户自己配 `sudo service docker start`——把环境 / 签名 / 依赖的"最后一公里"完全甩给开发者。
- 高频失败域与预判吻合：Android 签名失效（apksigner 35+ 版本突变）、Docker 基础镜像版本冲突、Xcode 依赖断裂，均属"知识密集、跨端差异大、Agent 无决策依据"的盲区。

### 对 ForgeKit 的启示
- **需求侧验证通过**：痛点普遍、致命、且主流 Agent 无原生解法 → "Agent 交付契约层"确实切中命门。
- **直接指导 v0.1 范围**：数据证明"窄 + 可验证 + 强制先规划"是压制复利错误率的正确策略 → 支持"先 Docker 硬闭环、deb 可选、强制 `plan_not_found`"。

## 2. 方向二：AI Agent 协议生态与 MCP 集成标杆

**调研目标**：决定 ForgeKit 技术产品形态——独立 CLI 还是标准 MCP Server，以及能否低成本寄生生态。

### 调研结论：MCP 已是标准，但"交付契约层"是无人区

| 发现 | 证据 |
|------|------|
| MCP 成事实标准 | Docker MCP Catalog 已含 **300+** Server；Claude Code / Cline / Cursor / Codex / Windsurf 全部原生支持 MCP |
| **关键空白** | 300+ Server 中**没有任何一个**构建 / 打包 / 发布工程专属 Server |
| 现有 Server 是 API 转发器 | GitHub MCP = 读仓库 / 管 Issue / PR（平台 API 转发，无 build/release）；Docker MCP = 容器管理；New Relic / Stripe / Grafana / Notion / Linear 均为外部服务集成 |
| 缺"决策与观测" | 现有 Server 只做命令 / API 转发，不提供强制计划、决策依据、结构化结果回溯 |

**ForgeKit 的楔子**：`Forge.md`（契约）+ `decision-rules.yaml`（约束）+ `result.json` / `plan_correction`（观测回溯）三件套，恰好填补现有 MCP 生态缺失的"决策 + 观测"能力。

### 对 ForgeKit 的启示
- **形态拍板**：坚定做**标准 MCP Server**（AGENT_INTEGRATION.md 已有决策被生态证实，无需 CLI 对冲）。
- **定位升级**：不是"又一个打包 MCP"，而是 **the delivery-contract MCP server**——在 300+ 转发器里，唯一带"决策 + 观测"的交付层。
- **寄生成本最低**：Ship 一个 MCP Server，即被所有主流 Agent 一键调用，无需逐家对接。

## 3. 方向三：企业级 B2D 场景的 AI 交付合规与验证需求

**调研目标**：验证企业是否愿为"可解释、安全可控的 AI 交付审计链"买单。

### 调研结论：合规压力正在形成付费意愿，且正对 ForgeKit 卖点

| 发现 | 证据 |
|------|------|
| 合规强制 | EU AI Act **2026-08-02** 全面强制执行；**82%** 企业已有 AI 治理顾虑（zylos 2026） |
| 人工校验诉求 | **63%** 企业领导要求校验所有 Agent 输出（静默幻觉驱动，Deloitte 2025） |
| 供应链可验证成标配 | Docker MCP 目录**强制 SBOM + provenance** 元数据才能上架 |
| 企业顾虑 = ForgeKit 卖点 | 怕 AI 乱改配置 / 引入漏洞 / 交付不可控 → `decision-rules`（Semantic Gateway 拦禁 `sudo` / 等价提权）+ `result.json`（审计链）+ HITL 触发器（信心 < 70% 挂起）直击 |

### 对 ForgeKit 的启示
- **商业化逻辑成立**：可解释、可审计的交付链是企业付费点，对应 `REQUIREMENTS.md` 的 L2（云端构建 / 签名 / 分发）与 L3（标准效应）。
- **属 L2/L3 叙事，不纳入 v0.x**：v0.x 本地优先是获客层，方向三的需求是"未来护城河"的证据，不宜在 v0.1 投入。

## 4. 综合研判与落地动作

**牵头方向 = 方向二**。理由：
1. 它消歧义、定形态（唯一拍板"做 MCP Server 而非 CLI"）。
2. 它暴露可防守的空白（300+ 服务器中"交付契约层"是无人区），给 ForgeKit 一个 first-mover 叙事与清晰差异化（转发器 vs 决策 + 观测）。
3. 它与现有资产精准对齐：`Forge.md` / `decision-rules.yaml` / `result.json` 不是凭空设计，而是正好补上现有 MCP 生态缺失的"决策与观测"能力——调研点名了独特卖点。
4. 寄生成本最低、见效最快（一个 MCP Server 即可被所有主流 Agent 调用），契合 v0.1 "跑通最小闭环"的节奏。

**注意**：方向一和方向三不是被放弃，而是作为**上游证据**与**下游商业化叙事**并入方向二的框架。

**落到 v0.1 的四动作**（详见 REQUIREMENTS.md §1 / §6 / §8）：
1. 定位陈述升级为"面向 AI Agent 的可验证交付契约层"。
2. 固化形态：只做 MCP Server，不在 CLI 上分心。
3. 差异化锚点：主打"决策 + 观测"对比 API 转发型 MCP，把契约三件套作为招牌能力讲透。
4. 范围不变、理由更硬：v0.1 继续"Docker 硬闭环 + deb 可选 + 服务器端优先"——方向一数据证明"窄 + 强制先规划"是压制复利错误率的正确策略。
