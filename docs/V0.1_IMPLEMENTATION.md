# ForgeKit v0.1 实现路径（员工可执行）

> 版本：v0.1 执行稿
> 关联：[REQUIREMENTS.md §8](./REQUIREMENTS.md)（每届 MVP，v0.1 最高价值）、[DESIGN.md](./DESIGN.md)、[TECHNICAL_PLAN.md](./TECHNICAL_PLAN.md)、[AGENT_INTEGRATION.md](./AGENT_INTEGRATION.md)
> 目标读者：执行 v0.1 开发的工程师 / 技术负责人
> 核心约束：**v0.1 = 用最便宜的表面（服务器端 Docker）端到端证明"带决策与观测的可验证交付契约"**。范围必须窄，闭环必须硬。

---

## 0. 目标与验收（契约视角）

v0.1 不止"能打包"，而是证明契约闭环：

| 验收项 | 标准 |
|--------|------|
| MCP 可被 Agent 发现 | 客户端能 `ListTools` 出 4 个工具 |
| `inspect_project` | 识别语言、入口、已有打包配置 |
| `generate_packaging_plan` | 生成可读 `Forge.md`（含 Decisions 决策依据段） |
| `build_docker_image`（硬闭环） | 本地生成可运行 `image:tag` |
| `pack_deb`（可选） | 仅当目标为 Ubuntu + systemd 时生成 `.deb` |
| **契约强制** | 构建类工具缺 `plan_path` → 返回 `plan_not_found` |
| **决策 + 观测** | 每次构建返回 `decision_basis` 与 `result.json` |
| E2E 可复现 | 示例 Python 项目从源码生成产物 |

## 1. 前置条件与既有资产

- **环境**：Node ≥ 18；`npm install`（依赖已锁定：`@modelcontextprotocol/sdk ^0.5.0`、`zod`、`commander`；dev：`vitest`、`typescript`）。
- **运行时**：构建阶段需本地 Docker daemon 可用。
- **可复用的既有代码（不要重写）**：
  - `src/mcp-server/index.ts` —— 已可用的 MCP Server 骨架（StdioServerTransport、`ListTools`/`CallTool` handler），调用 `registerTools()`。
  - `src/mcp-server/tools/registry.ts` —— 工具注册占位，待扩展。
  - `src/systems/servers/ubuntu/decision-rules.yaml` —— 已存在的服务器端决策规则，供 `generate_packaging_plan` / `build_docker_image` 读取生成 `decision_basis`。
- **约定**：沿用现有 import 风格（如 `'./tools/registry.js'` 的 `.js` 扩展名 specifier）；`tsconfig` 已配 `@mcp-server/*` `@capabilities/*` `@knowledge/*` 路径别名。

## 3. 里程碑拆解

### M0 · 启动与定位对齐（0.5 天）
- 确认 `npm run build`（tsc）与 `npm test`（vitest）可运行。
- 若 `tsc` 把 `tests/` 也编入 `dist`，在 `tsconfig.json` 的 `include` 移除 `tests/**/*`（测试交给 vitest，避免污染 `dist`）。
- （可选）将 `package.json` 的 `description` / `keywords` 改为"可验证交付契约层"口径，统一对外叙事。
- **验收**：`npm run build` 通过，占位 Server 可启动。

### M1 · 契约底座：类型 + Schema + 工具注册（2 天）
- 新建 `src/capabilities/types.ts`：统一结果结构 `{ status, artifacts, logs, warnings, decision_basis, next_actions }` + 错误结构 `{ code, summary, detail_log?, suggested_fix }`。
- 新建 `src/mcp-server/tools/schemas.ts`：用 `zod` 定义 4 工具 input/output Schema（对齐 DESIGN §5 / AGENT_INTEGRATION §6）。
- 扩展 `registry.ts`：注册 `inspect_project`、`generate_packaging_plan`、`build_docker_image`、`pack_deb`（可选），声明各自是否要求 `plan_path`。
- 新建 `executor.ts`：`CallTool` 按 `name` 路由到 capability；**构建类工具先校验 `plan_path`，缺失即返回 `plan_not_found`**。
- **验收**：MCP 客户端能列出 4 个工具；调用返回结构化内容（不再是 "pending" 占位）。

### M2 · `inspect_project`（1.5 天）
- 新建 `src/capabilities/inspect-project.ts`。
- 识别语言（Python 看 `pyproject.toml`/`requirements.txt`/`app.py`）、入口、`existing_packaging`（`Dockerfile`/构建配置）、`recommendations`。
- **验收**：对 fixture 输出正确结构化摘要；单测覆盖"无 Dockerfile""多入口"等分支。

### M3 · `generate_packaging_plan`（2 天）
- 新建 `src/packaging/forge-template.md`（依 `specs/PACKAGING_DOCUMENT.md` 的字段：Project / Goals / Build Strategy / Decisions / Risks / Results）。
- 新建 `src/capabilities/generate-packaging-plan.ts`：读 `decision-rules.yaml` → 渲染 `Forge.md` → 写入项目根目录；**决策依据来自兼容性矩阵**。
- 不覆盖用户手写内容（明确区域标记或先询问）。
- **验收**：fixture 生成可读 `Forge.md`，含 `Decisions` 段。

### M4 · `build_docker_image`（硬闭环，2.5 天）
- 新建 `src/capabilities/build-docker-image.ts`。
- 前置：**必须带 `plan_path`**，否则 `plan_not_found`。
- 读 `Forge.md` 配置 → 校验 `source_dir` / `Dockerfile`（`filesystem.ts` 做路径存在与越界检查）→ `docker build --platform linux/amd64`（`command.ts` 捕获日志）→ 采集 `image_ref` / `size` / `logs`。
- 输出：`status` + `artifacts` + `logs` + `warnings` + **`decision_basis`（来自 decision-rules）** + **`result.json`（`exit_code` / `stdout_snippet` / `state_delta`）**。
- 失败：结构化错误（`docker_daemon_unavailable` 等）+ `plan_correction` 提示，绝不抛原始异常字符串。
- **验收**：fixture 构建出可 `docker run` 的镜像；返回体含 `decision_basis` 与 `result.json`。

### M5 · `pack_deb`（可选，1.5 天）
- 新建 `src/capabilities/pack-deb.ts`；**仅当目标为 Ubuntu + systemd** 时实现，否则后置 v0.2。
- 生成 `debian/` 元数据（`control`/`rules`）→ `dpkg-deb` → `checksum.ts` 出 SHA256。
- **验收**：生成 `.deb` + 校验和；或明确后置并写入验收报告。

### M6 · 测试与 E2E（2 天）
- `tests/fixtures/sample-python-project/`：最小 Flask/Demo，含入口 + 依赖 + `Dockerfile`。
- `tests/unit/`：Schema、路径校验、错误解析。
- `tests/integration/`：executor → capability 路由。
- `tests/e2e/`：脚本模拟 Agent 调用 `inspect → plan → build` 完成闭环。
- **验收**：`npm test` 通过；一次手动 E2E 记录归档。

### M7 · 文档与验收（1 天）
- 更新 `README.md` 快速开始为真实可用命令。
- 产出验收报告，对照 REQUIREMENTS §9 成功指标（一次成功率 / 可解释率等，v0.1 以"闭环跑通 + 结构正确"为准）。
- **验收**：外部用户照文档可跑通。

## 4. 工具契约速查（实现时对齐）

| 工具 | 必需 `plan_path` | 关键输入 | 关键输出 |
|------|------------------|----------|----------|
| `inspect_project` | 否 | `source_dir` | `language`/`entrypoints`/`existing_packaging`/`recommendations` |
| `generate_packaging_plan` | 否 | `source_dir`,`goals`,`target_environment` | `plan_path`,`summary`,`warnings` |
| `build_docker_image` | **是** | `source_dir`,`plan_path`,`image_name`,`platform` | `image_ref`,`size_bytes`,`decision_basis`,`result.json` |
| `pack_deb` | **是** | `source_dir`,`plan_path`,`version`,`distro`,`arch` | `artifact_path`,`checksum`,`decision_basis` |

错误码至少包含：`plan_not_found`、`docker_daemon_unavailable`、`dockerfile_not_found`、`build_failed`、`invalid_path`。

## 6. 风险与门禁（沿用 TECHNICAL_PLAN §8/§9）

- 构建产物/生成文件放入 `dist/forgekit` 或临时目录，不污染用户项目。
- `Forge.md` 写入使用明确标记区域，避免覆盖用户手写内容。
- `Agent` 误传路径 → `filesystem.ts` 严格做存在性与越界检查。
- Docker 不可用 → 返回明确错误，不视为崩溃。
- 第一轮只做 M0–M1（协议 + 计划 + 接口稳定），**不要先写构建能力**（TECHNICAL_PLAN §10）。
