## 3. 可以保留的产品判断

### 核心假设：Docker 失败诊断

#3 描述的任务与当前方向最吻合：用户已经在构建，不需要先接受新的发布平台；价值来自从日志中指出证据、原因、修复动作和验证方法。该信号支持继续验证，但尚不能证明需求成立。

### 次级假设：可审查的自动打包

#1 和 #5 同时说明，自动化价值与信任风险必须一起处理。ForgeKit 可以保留计划预览、变更展示和构建验证，但近期不应把“AI 自动生成配置”作为唯一卖点，也不应默认修改或发布用户项目。

### 相邻假设：MCP 客户端配置

#2 指向安装与配置门槛。它与 ForgeKit 的采用路径直接相关，适合先用于完善自身的 `config show` / `config validate`、最短配置和错误提示。当前只有一条场景线索，不足以扩张为通用 MCP 配置管理产品。

### 非目标用户

#4 和 #5 说明两类低优先级人群：没有交付任务的本地研究项目，以及已有成熟发布平台且无明显失败成本的专家团队。后续招募应先筛选“最近 30 天有过 Docker 构建或接入失败”的参与者。

## 4. 当前产品决策

1. 保持“Docker 构建诊断与验证助手”为 v0.2 主线。
2. 把可解释性具体化为：错误类别、原始证据、可能原因、置信度、安全修复步骤、重试或验证条件。
3. 允许用户先诊断已有构建日志，降低必须先采用 ForgeKit 构建流程的门槛。
4. 把计划预览和无默认写操作作为信任机制，不把全自动发布列入 v0.2。
5. 把 ForgeKit 自身的 MCP 配置展示与校验列入接入迭代，不建设通用客户端管理器。
6. 在获得独立试点证据前，不恢复 deb/rpm、移动端、桌面端或云平台扩张。

具体交付顺序见 [产品迭代计划](../PRODUCT_ITERATION_PLAN.md)。

## 5. 仍未回答的问题

- 真实 BuildKit 日志中，高频失败类别到底是什么；
- ForgeKit 的建议是否比 Docker 原始提示更快、更准确；
- 用户是否愿意把失败日志交给本地 MCP 工具分析；
- 用户能否在 10 分钟内完成安装与客户端接入；
- 修复建议被采用后，构建是否真的成功；
- 用户是否在第二次失败时主动再次使用。

这些问题必须通过可复现失败样本、计时对照和独立用户试点回答，不能继续用问卷中的假设性描述代替。

## 6. 后续统计规则

- `问卷回答`：每条合规 Issue 计 1 条；
- `独立参与者`：按可区分的真实参与者计数，同一账号的多个角色不重复计数；
- `已访谈`：完成同步或异步追问，并获得一段可核验的过去行为描述；
- `愿意试用`：只记录意向，不计入成功门槛；
- `完成试用`：使用公开或脱敏项目完成约定任务并留下结果；
- `完成第二次使用`：首次试用后至少隔 1 天，对新的构建或失败再次使用。
