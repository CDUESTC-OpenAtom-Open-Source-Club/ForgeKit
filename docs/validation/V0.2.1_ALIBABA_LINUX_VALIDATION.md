# v0.2.1 Alibaba Cloud Linux 环境验证

> 验证日期：2026-07-22  
> 证据类型：项目方提供的远程环境功能验证  
> 统计边界：不计为独立用户访谈、真实项目构建或盲测准确率

## 1. 环境

| 项目 | 值 |
|---|---|
| 环境 | Alibaba Cloud ECS（完整公网 IP 不写入公开仓库） |
| 操作系统 | Alibaba Cloud Linux 3 / OpenAnolis |
| Node.js | v24.15.0 |
| 容器运行时 | Podman 4.9.4 |
| ForgeKit | v0.2.1 编译产物 |

## 2. 功能结果

| 用例 | 结果 |
|---|---|
| 模拟 Python/numpy 依赖冲突日志 | `pip_dependency_conflict`，`high`，通过 |
| 模拟 Dockerfile 语法错误日志 | `build_config_invalid`，`high`，通过 |

这两项证明编译产物能在该 Node/Linux 环境执行规则诊断，不证明真实 Docker/Podman 构建已经成功，也不用于计算准确率。

## 3. 安全结果

| 用例 | 结果 |
|---|---|
| 超过 1 MiB 的输入 | 返回 `log_too_large`，通过 |
| 项目内符号链接指向系统文件 | 路径越界被阻止，通过 |
| 密码、API Key、Token、用户路径 | 输出已脱敏，通过 |

验证时发现邮箱地址仍会保留。本轮已增加邮箱脱敏及自动回归测试，邮箱现在替换为 `[EMAIL]`。

## 4. 环境限制

服务器访问 Docker Hub 超时，因此未能完成真实镜像构建失败测试。`diagnose_build_failure` 本身不需要访问 Registry，仍可分析已有日志。后续真实构建验证应优先使用已缓存或内部可达的基础镜像，并单独记录镜像来源；不自动修改服务器镜像源。

## 5. 结论

本轮环境验证支持继续收集真实日志和开展独立用户试点。它证明了可移植性与基础安全控制，但不能单独证明 Issue #3 代表的用户需求已经成立，也不能替代 v0.2.1 的独立盲测和 v0.2.3 的重复使用指标。
