# ForgeKit 产品与商业策略

> 决策日期：2026-07-21  
> 决策主题：从“多平台能力规划”转为“可使用性优先、证据驱动扩展”。

## 1. 产品判断

ForgeKit 的核心竞争力不是执行更多构建命令，而是让 AI Agent 的交付过程具有四个属性：

- **有计划**：构建前明确目标、假设和风险；
- **受约束**：策略能够阻止明显错误或危险操作；
- **真验证**：不能以命令退出码代替产物真实存在和可运行；
- **可追溯**：产物可以关联到源码、规则、工具和环境。

因此，产品类别应定义为 **Agent-native verified delivery**，而不是通用打包器、Git 替代品或另一套 CI。

## 2. 当前真实瓶颈

当前仓库已经具备 MCP Server、计划生成、Docker 构建、deb 打包和测试基础。真正的瓶颈不是缺少 APK、IPA 或 exe，而是尚未证明：

- 非维护者是否能顺利安装和接入；
- 真实项目的成功率是否达到可依赖程度；
- 失败诊断是否能节省用户时间；
- 契约三件套是否比 Agent 直接运行 Shell 更有价值；
- 是否存在愿意持续使用甚至付费的团队。

所以接下来应把工程投入集中到安装体验、默认路径、诊断、样例、兼容性和真实用户验证。

## 3. 价值如何体现

不以代码量、平台数量或规则文件数量衡量价值，统一使用以下指标：

| 价值 | 指标 |
|------|------|
| 降低门槛 | 首次安装接入时间、无需人工修改次数 |
| 提高成功率 | 首次交付成功率、重试次数 |
| 节省时间 | 从源码到可运行产物的中位耗时 |
| 提高可解释性 | 可分类失败比例、建议被采用比例 |
| 提高可信度 | 具备完整来源和校验记录的产物比例 |
| 提高广泛性 | 真实项目、语言、架构、Agent/CI 组合覆盖数 |
| 验证商业性 | 活跃团队、留存、付费试点和单次交付成本 |

公开案例应展示使用前后对比，而不是只展示成功截图：原流程耗时、原失败点、ForgeKit 决策、最终产物和可验证证据都应保留。

## 4. 用户获取策略

第一批用户不从“大型企业采购”开始，而从能够快速反馈的真实项目开始：

1. 社团、课程和开源小项目：验证安装和教学体验；
2. 使用 AI Coding Agent 的独立开发者：验证日常工作流价值；
3. 维护多个小服务的小团队：验证规则复用和发布审查价值；
4. Agent 平台和插件作者：验证 MCP 作为集成入口的广泛性。

每个试点必须记录项目类型、原交付方式、耗时、失败原因、成功次数和是否愿意再次使用。

## 5. 开源与商业的边界

### 永久适合开源的核心

- MCP 协议与本地执行；
- `Forge.md` / Release Manifest 基础规范；
- 通用项目识别和公开规则；
- 本地构建、校验和基础诊断；
- 社区适配器与公开测试样例。

开放核心可以降低采用成本，避免用户担心构建工具锁定，并帮助契约格式形成生态。

### 可以商业化的能力

- 组织策略、审批、角色和审计留存；
- 私有兼容规则与企业环境适配；
- 托管构建、缓存、签名和产物保管；
- SBOM、provenance、合规报告与政策执行；
- SLA、长期支持、私有部署与兼容认证。

商业价值来自团队治理和可信执行，不应通过人为削弱开源本地版本制造。

## 6. 商业验证门槛

满足以下条件前不建设完整 SaaS：

- 至少 10 个真实项目完成交付；
- 至少 5 名外部用户完成两次以上使用；
- 至少 3 个团队明确提出共同的治理、审计或托管需求；
- 至少 3 个团队愿意参与付费试点；
- 可以估算每次构建的计算、存储和支持成本。

满足后优先销售人工辅助的团队试点，再决定是否产品化云服务。

## 7. 90 天执行建议

### 第 1–30 天：打磨可用入口

- 冻结新平台开发；
- 跑通干净环境安装；
- 完成最短 MCP 配置和一个公开演示；
- 增加 preflight 与 Git/版本追溯信息；
- 招募首批 3 个真实项目。

### 第 31–60 天：扩大真实样本

- 根据试点修复默认值、错误分类和文档；
- 扩展到 Python、Node.js、Go 代表项目；
- 建立兼容性矩阵和失败样本库；
- 发布可复现的案例报告。

### 第 61–90 天：验证留存与付费问题

- 达到 10 个真实项目；
- 测量首次成功率、交付耗时和再次使用率；
- 访谈小团队的策略、审计、签名和托管需求；
- 根据证据决定进入 v0.3，或继续修复 v0.2 可用性。

90 天结束时必须做一次继续/收缩/转向决策，不能仅因为路线图存在就自动扩张。

## 8. 失败与转向条件

出现以下情况之一，应停止当前产品叙事并重新判断：

- 目标用户更愿意让 Agent 直接生成 CI/Dockerfile，不接受独立契约；
- ForgeKit 未显著提高成功率或缩短交付时间；
- 大多数真实需求集中在部署运营，而不是构建交付；
- 规则库维护成本持续高于用户获得的价值；
- 团队只愿意使用开源能力，且不存在共同的付费问题。

转向并不代表项目失败：规则、诊断和 Release Manifest 仍可沉淀为 Agent 发布插件、CI 验证器或供应链策略组件。
