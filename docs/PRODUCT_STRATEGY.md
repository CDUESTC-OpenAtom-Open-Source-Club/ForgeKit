# ForgeKit 产品与公共价值策略

> 决策日期：2026-07-21  
> 决策主题：从“Agent 打包平台”收敛为“Docker 构建诊断与验证助手”。

ForgeKit 当前作为社团公共开源项目维护，不以商业化或收费为目标。项目价值优先体现在：初学者能理解、教师能教学、开发者能复现、社区能共同维护。

## 1. 产品判断

ForgeKit 当前首先解决 Docker 构建失败难定位的问题，核心价值按以下顺序排列：

- **早发现**：构建前检查环境、路径、运行时和 Registry；
- **能解释**：失败返回原因、证据和可执行的下一步；
- **真验证**：不能以命令退出码代替镜像真实存在和可运行；
- **易集成**：Agent 可通过 MCP 获得稳定的结构化结果。

因此，近期产品类别定义为 **MCP-native Docker build doctor**，而不是通用打包器、Docker 管理器、Git 替代品或另一套 CI。`Forge.md` 和 Release Manifest 继续作为安全与证据机制，但不再作为首要需求主张。

## 2. 当前真实瓶颈

当前仓库已经具备 MCP Server、计划生成、Docker 构建、deb 打包和测试基础。真正的瓶颈不是缺少 APK、IPA 或 exe，而是尚未证明：

- 非维护者是否能顺利安装和接入；
- 真实项目的成功率是否达到可依赖程度；
- 失败诊断是否能节省用户时间；
- 诊断是否比用户直接阅读 Docker/BuildKit 日志更快、更清楚；
- 是否存在愿意持续使用并参与反馈或贡献的课程、社团和开源团队。

所以接下来应把工程投入集中到真实失败样本、诊断准确率、安装体验和真实用户验证。完整决策见 [DIRECTION_DECISION.md](./DIRECTION_DECISION.md)。

## 3. 价值如何体现

不以代码量、平台数量或规则文件数量衡量价值，统一使用以下指标：

| 价值 | 指标 |
|------|------|
| 降低门槛 | 首次安装接入时间、无需人工修改次数 |
| 提高成功率 | 首次交付成功率、重试次数 |
| 节省时间 | 从源码到可运行产物的中位耗时 |
| 提高可解释性 | 可分类失败比例、建议被采用比例 |
| 提高可信度 | 具备完整来源和校验记录的产物比例 |
| 提高广泛性 | 真实项目、语言、架构、Agent/CI 组合覆盖数 |
| 验证公共价值 | 活跃用户、重复使用、教学采用和外部贡献 |

公开案例应展示使用前后对比，而不是只展示成功截图：原流程耗时、原失败点、ForgeKit 决策、最终产物和可验证证据都应保留。

## 4. 用户获取策略

第一批用户从能够快速反馈的真实项目开始：

1. 社团、课程和开源小项目：验证安装和教学体验；
2. 使用 AI Coding Agent 的独立开发者：验证日常工作流价值；
3. 维护多个小服务的小团队：验证规则复用和发布审查价值；
4. Agent 平台和插件作者：验证 MCP 作为集成入口的广泛性。

每个试点必须记录项目类型、原交付方式、耗时、失败原因、成功次数和是否愿意再次使用。

## 5. 开源与公共服务边界

### 永久适合开源的核心

- MCP 协议与本地执行；
- `Forge.md` / Release Manifest 基础规范；
- 通用项目识别和公开规则；
- 本地构建、校验和基础诊断；
- 社区适配器与公开测试样例。

开放核心可以降低采用成本，避免用户担心构建工具锁定，并帮助契约格式形成生态。

### 社团长期服务候选

- 公共教学案例、课程材料和新手工作坊；
- 社团共享的兼容规则与真实失败案例库；
- 公共构建演示、SBOM 与 provenance 教学；
- 面向开源项目的发布规范辅导；
- 社区贡献者维护的语言和平台适配器。

公共价值来自降低交付知识门槛，不应通过增加账户、收费墙或复杂平台削弱本地开源版本。

## 6. 公共价值验证门槛

当前不建设完整 SaaS。项目是否继续扩展，使用以下公共价值证据判断：

- 至少 10 个真实项目完成交付；
- 至少 5 名外部用户完成两次以上使用；
- 至少 3 个课程、社团或开源项目提出共同需求；
- 至少 3 名外部贡献者参与案例、规则或代码维护；
- 文档能够被初学者独立理解和复现。

满足后优先扩展教学材料、兼容矩阵和社区协作，不以云服务产品化作为默认终点。

## 7. 90 天执行建议

### 第 1–30 天：打磨可用入口

- 冻结新平台开发；
- 跑通干净环境安装；
- 完成最短 MCP 配置和一个公开演示；
- 增加 preflight 与 Git/版本追溯信息；
- 招募首批 3 个真实项目。

### 第 31–60 天：扩大真实样本

- 根据试点修复默认值、错误分类和文档；
- 扩展到 Python、Node.js、Go 代表项目；
- 建立兼容性矩阵和失败样本库；
- 发布可复现的案例报告。

### 第 61–90 天：验证留存与社区参与

- 达到 10 个真实项目；
- 测量首次成功率、交付耗时和再次使用率；
- 访谈课程、社团和小团队的教学、审查与兼容性需求；
- 根据证据决定进入 v0.3，或继续修复 v0.2 可用性。

90 天结束时必须做一次继续/收缩/转向决策，不能仅因为路线图存在就自动扩张。

## 8. 失败与转向条件

出现以下情况之一，应停止当前产品叙事并重新判断：

- 目标用户更愿意让 Agent 直接生成 CI/Dockerfile，不接受独立契约；
- ForgeKit 未显著提高成功率或缩短交付时间；
- 大多数真实需求集中在部署运营，而不是构建交付；
- 规则库维护成本持续高于用户获得的价值；
- 用户只做一次性尝试，不愿重复使用、反馈或参与共同维护。

转向并不代表项目失败：规则、诊断和 Release Manifest 仍可沉淀为 Agent 发布插件、CI 验证器或供应链策略组件。
