# 项目方向决策：从“Agent 打包平台”收敛到 Docker 构建诊断

> 决策日期：2026-07-21  
> 状态：立即生效，进入需求验证阶段。

## 结论

ForgeKit 的近期定位调整为：

> **面向初学者和 AI Agent 的 Docker 构建诊断与验证助手。**

它帮助用户在构建前发现问题、在构建失败后得到可操作解释，并验证最终镜像确实可以运行。MCP 是主要集成入口，但不是产品价值本身。

## 为什么调整

现有材料能支持两个判断：Docker 构建失败是持续出现的真实问题，MCP 用户确实会操作 Docker。材料尚不能证明用户主动需要“AI Agent 专用打包平台”，也不能证明强制生成计划文件本身构成需求。

需要谨慎解释这些证据：GitHub Issue 数量和竞品 Star 只能证明问题与关注存在，不能等同于迁移意愿、活跃用户或 ForgeKit 的采用需求。最终判断必须来自真实试点。

## 新的价值排序

1. **发现问题**：构建前检查 Docker、目录、入口、磁盘和 Registry；
2. **解释失败**：把 Docker 构建输出转换为原因、证据和下一步；
3. **验证结果**：镜像存在、可以启动、健康检查通过；
4. **方便 Agent 调用**：通过 MCP 返回稳定的结构化结果；
5. **保留交付证据**：计划和 Manifest 作为高级保障，不作为获客主张。

## 功能处置

| 能力 | 决策 | 说明 |
|---|---|---|
| Preflight | 核心 | 用户无需开始构建即可获得诊断结果 |
| 错误诊断 | 核心 | 优先覆盖真实 Docker 失败样本 |
| Docker 构建与运行验证 | 核心 | 不只检查命令退出码 |
| MCP 接入 | 核心入口 | 与 Agent 集成，但不与 Docker 官方能力竞争 |
| 项目识别 | 支撑 | 为诊断提供上下文 |
| `Forge.md` | 保留、弱化宣传 | 当前构建安全门保持不变；验证用户价值后再决定是否可选 |
| Release Manifest | 高级/可选价值 | 不作为个人初学者的首要理由 |
| deb、arm64、多平台 | 暂停扩张 | 只有真实试点提出共同需求才恢复 |

## 与现有 Docker MCP 工具的边界

ForgeKit 不定位为“远程控制 Docker 的完整 MCP Server”，不优先增加容器生命周期、端口、网络、Volume、Secret 管理。它专注构建路径的检查、解释和验证，可以与 Docker 操作型 MCP 工具互补。

## 30 天验证任务

- 收集至少 30 个公开、可复现的 Docker 构建失败样本；
- 将失败归为环境、路径、Dockerfile、依赖、Registry、架构和权限等类别；
- 为高频类别增加回归样本与结构化建议；
- 让至少 5 名非维护者用自己的项目完成一次诊断；
- 对比“直接看 Docker 日志”和“使用 ForgeKit”定位问题所需时间；
- 记录用户是否愿意第二次使用。

## 继续、调整或停止的门槛

- 若至少 60% 的试点用户认为诊断缩短了定位时间，继续建设诊断规则和项目覆盖；
- 若用户只需要静态 Dockerfile 检查，收缩为 hadolint/BuildKit 输出的解释层；
- 若用户主要需要部署和容器管理，考虑与现有 Docker MCP 项目集成，而非复制其功能；
- 若 30 天内无法获得 5 名真实用户或重复使用，冻结产品功能开发，仅保留教学与技术成果。
